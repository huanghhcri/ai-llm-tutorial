## 一、章节概述

### 本章学什么

本章将量化后的模型部署到养老院的服务器上，构建**完整的私有化 AI 服务**。你将掌握：

- **Ollama 部署**：最简单的大模型本地运行方案（一行命令）
- **vLLM 部署**：高性能推理引擎（支持并发、PagedAttention）
- **FastAPI 封装**：将模型包装为 OpenAI 兼容 API
- **Docker 容器化**：一键部署和扩展
- **GPU 规划**：如何选择合适的硬件
- **养老院实战**：完整的私有化部署方案

### 为什么学

养老院的长者健康数据是**高度敏感的个人信息**，不能发送到外部 API（如 OpenAI、Qwen Cloud）。私有化部署让所有数据处理都在养老院内部完成，满足数据合规要求。

此外，私有化部署的**长期成本更低**——一次购买硬件，持续使用无额外费用。

### 2.2 Ollama 部署（最简方案）

#### 类比

Ollama 就像"一键安装包"——下载、安装、运行，三步搞定。对标 C# 的 `dotnet run`。

```bash
# ========== 1. 安装 Ollama ==========
# Linux/macOS
curl -fsSL https://ollama.com/install.sh | sh

# Windows
# 从 https://ollama.com/download 下载安装包

# ========== 2. 拉取模型 ==========
# 拉取预量化好的 GGUF 模型
ollama pull qwen3:8b           # Qwen3-8B（自动选择合适精度）
ollama pull qwen3:8b-q4_K_M    # 指定 Q4_K_M 量化

# ========== 3. 运行模型 ==========
ollama run qwen3:8b "高血压长者怎么护理？"

# ========== 4. 启动 API 服务 ==========
# Ollama 默认在 11434 端口提供 OpenAI 兼容 API
ollama serve

# 测试 API
curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen3:8b",
    "messages": [
      {"role": "system", "content": "你是养老院健康顾问。"},
      {"role": "user", "content": "张大爷血压160/100怎么处理？"}
    ],
    "temperature": 0.3,
    "max_tokens": 500
  }'
```

#### 导入自定义模型（微调后的养老院模型）

```bash
# 1. 将微调后的模型转为 GGUF（第 14 章内容）
# 2. 创建 Modelfile
cat > Modelfile << 'EOF'
FROM ./nursing_model_q4_k_m.gguf

SYSTEM """你是养老院的智能健康评估助手"小护"。
根据长者的生命体征数据，给出专业的健康评估和护理建议。

📊 情况分析

💡 处理建议

PARAMETER top_p 0.9
PARAMETER num_ctx 4096
PARAMETER stop "###"
EOF

# 3. 创建 Ollama 模型
ollama create nursing-assistant -f Modelfile

# 4. 测试
ollama run nursing-assistant "张大爷血压165/105，血氧88%，怎么处理？"

# 5. API 调用
curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "nursing-assistant", "messages": [{"role": "user", "content": "血压高怎么办？"}]}'
```

### 2.4 FastAPI 自定义封装

```python
# 安装：uv add fastapi uvicorn llama-cpp-python
# 版本：fastapi 0.139.0, uvicorn 0.51.0, llama-cpp-python 0.3.33
#
# 适用场景：需要在 API 层添加自定义逻辑（如 RAG 检索、权限校验、日志记录）

from fastapi import FastAPI, HTTPException
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from typing import Optional
import json
import time
import uvicorn

app = FastAPI(title="养老院 AI 服务", version="1.0.0")


# ========== 数据模型（对标 OpenAI API 格式）==========

class ChatMessage(BaseModel):
    role: str          # "system" / "user" / "assistant"
    content: str

class ChatRequest(BaseModel):
    model: str
    messages: list[ChatMessage]
    temperature: float = 0.3
    max_tokens: int = 1000
    stream: bool = False

class ChatResponse(BaseModel):
    id: str
    object: str = "chat.completion"
    created: int
    model: str
    choices: list[dict]
    usage: dict


# ========== 模型加载 ==========

# 全局模型实例（启动时加载一次）
# from llama_cpp import Llama
# llm = Llama(
#     model_path="./nursing_model_q4_k_m.gguf",
#     n_ctx=4096,
#     n_gpu_layers=-1,      # -1=全部加载到GPU，0=纯CPU
#     n_threads=4,
#     verbose=False,
# )

# 模拟模型（实际运行时替换为真实模型）
class MockLLM:
    def create_chat_completion(self, messages, max_tokens=500, temperature=0.3, stream=False):
        last_msg = messages[-1]["content"] if messages else ""
        response_text = f"📊 **分析**：根据您描述的情况...\n⚠️ **风险**：需关注\n💡 **建议**：请咨询主治医生确认。"
        if stream:
            return self._stream_response(response_text)
        return {
            "choices": [{"message": {"role": "assistant", "content": response_text}, "finish_reason": "stop"}],
            "usage": {"prompt_tokens": 100, "completion_tokens": 50, "total_tokens": 150},
        }
    
    def _stream_response(self, text):
        for char in text:
            yield {"choices": [{"delta": {"content": char}, "finish_reason": None}]}
            time.sleep(0.02)

llm = MockLLM()


# ========== API 端点 ==========

@app.post("/v1/chat/completions", response_model=None)
async def chat_completions(request: ChatRequest):
    """
    OpenAI 兼容的 Chat Completions API。
    ASP.NET Core 后端可以直接用 HttpClient 调用。
    """
    try:
        messages = [{"role": m.role, "content": m.content} for m in request.messages]
        
        if request.stream:
            # 流式输出
            def generate():
                for chunk in llm.create_chat_completion(
                    messages=messages,
                    max_tokens=request.max_tokens,
                    temperature=request.temperature,
                    stream=True,
                ):
                    yield f"data: {json.dumps(chunk)}\n\n"
                yield "data: [DONE]\n\n"
            
            return StreamingResponse(generate(), media_type="text/event-stream")
        
        # 非流式输出
        result = llm.create_chat_completion(
            messages=messages,
            max_tokens=request.max_tokens,
            temperature=request.temperature,
            stream=False,
        )
        
        return ChatResponse(
            id=f"chatcmpl-{int(time.time())}",
            created=int(time.time()),
            model=request.model,
            choices=result["choices"],
            usage=result.get("usage", {}),
        )
    
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))


@app.get("/v1/models")
async def list_models():
    """列出可用模型"""
    return {
        "data": [
            {"id": "nursing-assistant", "object": "model", "owned_by": "nursing-home"},
        ]
    }


@app.get("/health")
async def health_check():
    """健康检查"""
    return {"status": "ok", "model": "nursing-assistant"}


# 启动命令：uvicorn main:app --host 0.0.0.0 --port 8000
if __name__ == "__main__":
    uvicorn.run(app, host="0.0.0.0", port=8000)
```

### 2.5 Docker 容器化部署

```dockerfile
# Dockerfile — Ollama 方案
# 基于 Ollama 官方镜像，一键部署

FROM ollama/ollama:latest

# 复制模型文件和启动脚本
COPY nursing_model_q4_k_m.gguf /models/
COPY Modelfile /models/
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

# 暴露端口
EXPOSE 11434

# 使用 entrypoint 脚本：先启动 Ollama，再创建模型，最后保持运行
ENTRYPOINT ["/entrypoint.sh"]
```

`entrypoint.sh` 内容：
```bash
#!/bin/bash
# 启动 Ollama 后台服务
ollama serve &
# 等待服务就绪
sleep 3
# 创建模型
ollama create nursing-assistant -f /models/Modelfile
# 保持前台运行
wait
```

## 三、养老院业务实战案例

## 四、知识点总结

| 概念 | 一句话解释 | 关键要点 |
|------|-----------|---------|
| **私有化部署** | 模型运行在自己的服务器上 | 数据不出院，满足合规要求 |
| **Ollama** | 最简单的大模型本地运行方案 | 一行命令安装，OpenAI 兼容 API |
| **vLLM** | 高性能 LLM 推理引擎 | PagedAttention + 连续批处理，最强并发 |
| **FastAPI** | Python Web 框架 | 自定义 API 逻辑，灵活但需要更多开发 |
| **llama-cpp-python** | llama.cpp 的 Python 绑定 | CPU/GPU 混合推理，GGUF 格式 |

## 五、本章面试题

### 题目 1：为什么养老院需要私有化部署？有什么替代方案？

**难度**：⭐  
**类型**：需求分析

**参考答案**：

私有化部署的核心原因是**数据合规**——长者的健康数据（血压、病史、用药）属于敏感个人信息，根据《个人信息保护法》和《健康医疗大数据应用管理暂行办法》，健康数据原则上不能出境或发送到第三方服务器。使用公共 API（如 OpenAI、Qwen Cloud）意味着数据会发送到外部服务器，存在合规风险。替代方案：① **国内合规 API**——使用通过数据安全评估的国内 API（如阿里云、百度云的私有化部署方案），数据在国内但不在院内；② **混合方案**——非敏感任务（如护理知识问答）用公共 API，敏感任务（如病历分析）用本地模型；③ **私有云**——在云服务商的私有化环境中部署模型（如阿里云专有云），数据不出租户。养老院推荐：纯私有化部署（Ollama/vLLM），数据完全在院内。

### 题目 2：Ollama 和 vLLM 分别适合什么场景？如何选择？

**难度**：⭐⭐  
**类型**：技术选型

**参考答案**：

| 维度 | Ollama | vLLM |
|------|--------|------|
| 部署难度 | 极低（一行命令） | 中等（需要 Python 环境） |
| 模型格式 | GGUF | HF/AWQ/GPTQ |
| 推理方式 | CPU/GPU 混合 | 纯 GPU |
| 速度 | 中等 | 最快（PagedAttention） |
| 并发 | 有限（3-5） | 强（10-20+） |
| 资源占用 | 低 | 较高 |
| 适用场景 | 本地开发、单人/小团队 | 生产环境、多用户 |

选择建议：① 养老院 1-3 位护理员使用 → Ollama（最简单，RTX 4060 可运行）；② 养老院 5+ 位护理员同时使用 → vLLM（并发能力强）；③ 没有 GPU → Ollama + GGUF（CPU 推理）；④ 需要自定义逻辑（RAG、权限） → FastAPI + llama-cpp-python。

### 题目 3：如何将 AI 服务集成到 ASP.NET Core 后端？

**难度**：⭐⭐  
**类型**：系统集成

**参考答案**：

集成步骤：① **配置**——在 `appsettings.json` 中添加 AI 服务的 URL 和模型名；② **注册 HttpClient**——在 `Program.cs` 中用 `AddHttpClient` 注册 AI 服务客户端；③ **封装服务类**——创建 `AiService` 类，封装 Chat Completions API 调用；④ **流式支持**——如果需要流式输出，使用 `HttpClient.SendAsync` + `ReadAsStreamAsync`，通过 SignalR 推送给前端；⑤ **错误处理**——设置合理的超时（60-120 秒），处理连接失败和模型加载中的情况；⑥ **依赖注入**——将 `AiService` 注册为 Scoped 服务，在 Controller 中注入使用。关键：AI 服务和 ASP.NET Core 之间的通信走内网 HTTP，不需要 HTTPS（内网信任环境）。

### 题目 4：Docker 部署有什么优势？如何用 Docker Compose 编排 AI 服务？

**难度**：⭐⭐  
**类型**：部署实践

**参考答案**：

Docker 的优势：① **环境隔离**——模型运行环境与宿主机隔离，不影响其他服务；② **一键部署**——`docker compose up -d` 启动所有服务；③ **易于扩展**——需要更多并发时，启动多个容器实例；④ **版本管理**——镜像版本化，可以快速回滚。Docker Compose 编排包含三个服务：① **ai-model**——Ollama 服务，挂载 GPU，持久化模型数据；② **nursing-api**——ASP.NET Core 后端，通过环境变量配置 AI 服务地址；③ **nginx**——反向代理，统一入口，SSL 终止。关键配置：GPU 直通（`deploy.resources.reservations.devices`）、健康检查（`healthcheck`）、数据卷持久化（`volumes`）。

### 题目 5：如何规划养老院的 GPU 硬件？

**难度**：⭐  
**类型**：成本规划

**参考答案**：

规划原则：根据**模型大小**和**并发用户数**选择 GPU。① **7B 模型 INT4**——需要 4GB 显存，RTX 4060 Ti（8GB，¥3,000）即可，剩余显存用于 KV Cache 和并发缓冲；② **14B 模型 INT4**——需要 8GB 显存，RTX 4070 Ti Super（16GB，¥5,500）；③ **70B 模型 INT4**——需要 35GB 显存，需要 A100（80GB）或 2×RTX 4090。养老院推荐：① 预算极低 → 纯 CPU（Ollama + GGUF，无额外硬件成本，速度慢但可用）；② 预算 ¥3,000 → RTX 4060 Ti（7B 模型流畅运行）；③ 预算 ¥12,000 → RTX 4090（14B 模型或多并发）。长期成本：电费约 ¥100-200/月，无其他费用。

### 题目 6：什么是 PagedAttention？为什么 vLLM 比 Ollama 快？

**难度**：⭐⭐⭐  
**类型**：技术原理

**参考答案**：

PagedAttention 是 vLLM 的核心创新，解决了 LLM 推理中的**显存碎片化**问题。传统推理中，每个请求需要预分配最大长度的 KV Cache 显存（如 4096 tokens），但实际生成可能只有 200 tokens，大量显存被浪费。PagedAttention 借鉴了操作系统的**虚拟内存分页**思想——将 KV Cache 分成固定大小的"页"，按需分配，用完释放。效果：① 显存利用率提升 2-4 倍；② 支持更大并发（更多请求共享同一块显存）；③ 吞吐量提升 2-4 倍。vLLM 还使用**连续批处理**——不等一个请求处理完再处理下一个，而是在每个生成步骤后动态插入新请求，GPU 利用率更高。这就是 vLLM 比 Ollama 快的核心原因。

### 题目 7：如何监控 AI 服务的运行状态？

**难度**：⭐  
**类型**：运维

**参考答案**：

监控维度：① **GPU 使用率**——用 `nvidia-smi` 或 Prometheus + DCGM Exporter 监控 GPU 利用率、显存占用、温度；② **API 延迟**——记录每次 API 调用的响应时间（P50/P95/P99），超过阈值告警；③ **吞吐量**——每秒处理的请求数（QPS）和生成的 token 数；④ **错误率**——API 调用失败的比例，包括超时、模型错误等；⑤ **健康检查**——定期调用 `/health` 端点，确认服务存活。工具推荐：① 简单方案 → 日志文件 + 定时检查脚本；② 专业方案 → Prometheus + Grafana（可视化仪表盘）；③ 告警 → 当 GPU 温度 >85°C 或 API 延迟 >10s 时发送通知。

### 题目 8：如何处理模型服务的高可用和故障恢复？

**难度**：⭐⭐  
**类型**：可靠性

**参考答案**：

① **自动重启**——用 systemd（Linux）或 Docker restart policy 确保服务崩溃后自动重启；② **健康检查**——配置负载均衡器（Nginx）的健康检查，自动剔除不健康的实例；③ **多实例部署**——部署 2 个 Ollama 实例，用 Nginx 做负载均衡，一个挂了另一个继续服务；④ **模型预热**——服务启动后先发一个测试请求，确保模型加载完成后再接收流量；⑤ **优雅降级**——当 AI 服务不可用时，ASP.NET Core 后端返回"AI 服务暂时不可请稍后重试"，而不是整个系统崩溃；⑥ **数据持久化**——Ollama 的模型数据用 Docker Volume 持久化，容器重建不丢失模型。

### 题目 9：如何优化 AI 服务的响应速度？

**难度**：⭐⭐  
**类型**：性能优化

**参考答案**：

① **使用量化模型**——INT4 比 FP16 推理速度快 1.5-2 倍；② **GPU 部署**——GPU 比 CPU 快 5-10 倍；③ **流式输出**——用 stream 模式，用户第一个字在 200ms 内就能看到，感知延迟大幅降低；④ **减少 max_tokens**——限制输出长度，避免生成过长内容；⑤ **Prompt 优化**——精简 System Prompt，减少输入 token 数；⑥ **缓存**——对相同或相似的请求缓存结果（如常见护理问题的回答）；⑦ **选择更快的模型**——简单任务用小模型（3B），复杂任务才用大模型（14B）；⑧ **vLLM 的连续批处理**——高并发时比 Ollama 快 2-4 倍。
