## 10.1 为什么需要认证授权

### 10.1.1 从养老院的门禁说起

想象一家没有门禁的养老院：凌晨2点陌生人走进来，没人拦他——这就是**没有认证**。保洁阿姨通过了门禁（认证通过）却走进药房取药——这就是**授权失败**。家属张女士来探望父亲，前台确认身份后发了访客证，但她只能进她父亲的房间——这就是**基于角色的授权**。

| 概念 | 英文 | 含义 | 养老院类比 |
|------|------|------|-----------|
| 认证 | Authentication | 你是谁？ | 前台确认你的身份 |
| 授权 | Authorization | 你能做什么？ | 确认你能进入哪些区域 |

### 10.1.2 从无保护到有保护

```csharp
// 无保护——任何人都能访问
[Area("yls")]
[Route("api/yls/elderly")]
public class ElderlyController : AbpController
{
    [HttpGet("{id}/health-records")]
    public async Task<List<HealthRecordDto>> GetHealthRecords(Guid id)
    {
        return await _elderlyAppService.GetHealthRecordsAsync(id);
    }
}

// 有保护——必须登录且有权限
[Authorize]
[Area("yls")]
[Route("api/yls/elderly")]
public class ElderlyController : AbpController
{
    [HttpGet("{id}/health-records")]
    [Authorize("HealthRecord.Read")]
    public async Task<List<HealthRecordDto>> GetHealthRecords(Guid id)
    {
        return await _elderlyAppService.GetHealthRecordsAsync(id);
    }
}
```

## 10.2 OAuth2 四种授权模式

### 10.2.1 授权码模式 —— 正式访客登记（最安全，推荐）

家属张女士探望父亲：前台验证身份证 → 发临时访客单（授权码） → 门禁处换正式访客证（Token）。

技术流程：用户点击"微信登录" → 跳转微信授权页 → 用户同意 → 微信返回 code → 后端用 code 换 Token。**授权码只能用一次且经后端交换，Token 不暴露在浏览器。**

```
┌──────────┐                    ┌──────────┐
│  浏览器   │ ──1.点击登录──→    │  微信服务器 │
│          │ ←─2.重定向授权页──  │          │
│          │ ──3.用户同意──→     │          │
│          │ ←─4.返回code──     │          │
│          │                    └──────────┘
│          │ ──5.code发给──→  ┌──────────┐
└──────────┘                  │  后端服务器 │
                              │ 用code换Token│
                              └──────────┘
```

### 10.2.2 密码模式 —— 员工刷卡

护工小李直接用工号密码登录，系统验证后发放通行令牌。适用于自家第一方应用（完全可信的客户端）。

```csharp
var request = new
{
    grant_type = "password",
    username = "nurse_zhang",
    password = "Abc123456",
    client_id = "nursing-home-app",
    scope = "openid profile"
};
```

### 10.2.3 客户端模式 —— 快递员取件

快递员不需要个人身份，只出示快递公司工牌。用于服务间通信，不涉及用户身份——如养老院的健康监测设备自动上报数据。

```csharp
// 养老院监控设备自动上报数据（无需用户身份）
var request = new
{
    grant_type = "client_credentials",
    client_id = "monitoring-device-001",
    client_secret = "device-secret",
    scope = "monitoring-api"
};
```

### 10.2.4 隐式模式 —— 临时通行证（已废弃）

Token 直接暴露在 URL 中，安全性最低。OAuth 2.1 已废弃，新项目应使用授权码模式 + PKCE 替代。

| 模式 | 安全性 | 适用场景 | 推荐 |
|------|--------|---------|------|
| 授权码模式 | 最高 | 第三方登录（微信） | ✅ |
| 密码模式 | 中等 | 第一方应用 | ✅ |
| 客户端模式 | 中等 | 服务间通信 | ✅ |
| 隐式模式 | 最低 | 单页应用（已废弃） | ❌ |

## 10.3 JWT Token 结构（面试常问）

### 10.3.1 三段式结构

JWT 由 `.` 分隔的三段组成：

```
eyJhbG...sR8U
|____Header____|____Payload____|____Signature____|
```

**Header（头部）**：声明算法和类型。类比：访客证的卡片材质和防伪技术说明。

```json
{ "alg": "HS256", "typ": "JWT" }
```

**Payload（载荷）**：存放用户信息（ID、角色、权限、过期时间）。类比：访客证上的姓名和有效期。**注意：Payload 只是 Base64Url 编码，不是加密，不要放密码等敏感信息！**

```json
{
  "sub": "user-001",
  "name": "张护士",
  "role": "Nurse",
  "exp": 1700000000
}
```

**Signature（签名）**：`HMACSHA256(base64(header) + "." + base64(payload), secretKey)`。类比：养老院的防伪印章，别人无法伪造。

### 10.3.2 签名验证原理

```csharp
public bool ValidateJwt(string token, string secretKey)
{
    var parts = token.Split('.');
    // 用同样的算法和密钥重新计算签名
    var expected = Base64UrlEncode(
        HMACSHA256(parts[0] + "." + parts[1], secretKey));
    // 对比签名是否一致
    return parts[2] == expected;
}
```

**核心：只有拥有密钥的服务端才能生成有效签名。篡改 Payload 则签名对不上。**

### 10.3.3 Base64Url 编码

与标准 Base64 的区别：`+` → `-`，`/` → `_`，去掉 `=` 填充。这样 JWT 可以安全地放在 URL 中传输。

## 10.4 JWT 过期与 RefreshToken 机制

### 10.4.1 双 Token 机制

| Token 类型 | 有效期 | 用途 | 养老院类比 |
|-----------|--------|------|-----------|
| Access Token | 30 分钟 | 访问 API | 当天通行令牌 |
| Refresh Token | 7 天 | 刷新 Access Token | 长期访客证 |

### 10.4.2 无感刷新流程

```
前端请求API → Access Token过期 → 401 Unauthorized
    → 前端拦截401 → 用Refresh Token换新Token
    → 用新Token重试原请求 → 用户无感知
```

```csharp
protected override async Task<HttpResponseMessage> SendAsync(
    HttpRequestMessage request, CancellationToken ct)
{
    request.Headers.Authorization =
        new AuthenticationHeaderValue("Bearer", await GetTokenAsync());
    var response = await base.SendAsync(request, ct);

    if (response.StatusCode == HttpStatusCode.Unauthorized)
    {
        var newToken = await RefreshTokenAsync();
        request.Headers.Authorization =
            new AuthenticationHeaderValue("Bearer", newToken);
        response = await base.SendAsync(request, ct);
    }
    return response;
}
```

### 10.4.3 Refresh Token 安全策略

Refresh Token 一次性使用，用过即销毁。如果有人重复使用已用的 Refresh Token（说明可能被盗），应立即吊销该用户的所有 Token。

## 10.5 Cookie vs Session vs Token（面试必问）

| 对比维度 | Cookie | Session | Token (JWT) |
|---------|--------|---------|-------------|
| 存储位置 | 浏览器端 | 服务端 | 客户端任意位置 |
| 携带方式 | 自动随请求发送 | 通过Cookie的Session ID | 手动放Header |
| 跨域支持 | ❌ 同源限制 | ❌ 依赖Cookie | ✅ 天然支持 |
| 服务端状态 | 无状态 | 有状态 | 无状态 |
| 扩展性 | 差 | 差（需共享Session） | 好（无需共享） |
| 移动端支持 | ❌ | ❌ | ✅ |
| 典型场景 | 传统Web | 传统Web | 前后端分离/微服务 |

**养老院类比**：Cookie=手上的访客贴纸（自动展示）；Session=前台登记簿（信息在服务端）；Token=独立电子门禁卡（自包含，到哪都能用）。

**Token 优势**：无状态、支持跨域、支持移动端、适合微服务和分布式架构。

## 10.6 IdentityServer4 在项目中的角色

IdentityServer4 是 .NET 生态的 OAuth2/OpenID Connect 服务器，相当于养老院的**前台+门禁+身份管理中心**。

**核心职责**：统一用户认证、Token 签发与验证、客户端管理、API 资源管理。

```csharp
// 配置 JWT Bearer 认证
services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://auth.nursing-home.com";
        options.Audience = "NursingHomeAPI";
    });
```

与 ABP 集成：ABP 内置了 IdentityServer4 支持，通过配置即可完成 OAuth2 服务器搭建、Token 签发、API 资源定义等。认证服务可独立部署，所有微服务信任同一认证中心。

```csharp
// 定义API资源和客户端
public class NursingHomeIdentityServerConfig
{
    public static IEnumerable<ApiResource> GetApiResources()
    {
        return new List<ApiResource>
        {
            new ApiResource("NursingHomeAPI", "养老院管理系统API")
            {
                Scopes = { "NursingHomeAPI" }
            }
        };
    }

    public static IEnumerable<Client> GetClients()
    {
        return new List<Client>
        {
            // 养老院Web端
            new Client
            {
                ClientId = "nursing-home-web",
                ClientSecrets = { new Secret("web-secret".Sha256()) },
                AllowedGrantTypes = GrantType.ResourceOwnerPassword,
                AllowedScopes = { "openid", "profile", "NursingHomeAPI" }
            },
            // 养老院家属端（微信登录）
            new Client
            {
                ClientId = "nursing-home-family",
                ClientSecrets = { new Secret("family-secret".Sha256()) },
                AllowedGrantTypes = GrantType.AuthorizationCode,
                AllowedScopes = { "openid", "profile", "NursingHomeAPI" },
                RedirectUris = { "https://family.nursing-home.com/callback" }
            }
        };
    }
}
```

```csharp
// 角色初始化种子数据
```

## 10.8 权限定义与使用

### 10.8.1 PermissionDefinitionProvider 定义权限

在模块的 `PermissionDefinitionProvider` 中集中定义所有权限（完整示例见本章实战案例）：

```csharp
public class NursingHomePermissionProvider : PermissionDefinitionProvider
{
    public override void Define(IPermissionDefinitionContext context)
    {
        // 老人管理权限组
        var elderly = context.AddGroup("Elderly", "老人管理");
        elderly.AddPermission("Elderly.Read", "查看老人信息");
        elderly.AddPermission("Elderly.Create", "新建老人档案");
        elderly.AddPermission("Elderly.Update", "编辑老人信息");

        // 健康管理权限组（含子权限）
        var health = context.AddGroup("Health", "健康管理");
        var healthRead = health.AddPermission("Health.Read", "查看健康档案");
        healthRead.AddChild("Health.Record.Read", "查看详细病历");
        health.AddPermission("Health.Write", "编辑健康档案");

        // 费用管理权限组
        var billing = context.AddGroup("Billing", "费用管理");
        billing.AddPermission("Billing.Read", "查看账单");
        billing.AddPermission("Billing.Create", "创建账单");
        billing.AddPermission("Billing.Confirm", "确认缴费");
    }
}
```

> 💡 此 Provider 需在模块的 `ConfigureServices` 中注册，且模块需通过 `[DependsOn]` 引用 ABP Identity 模块。

### 10.8.2 [Authorize] 特性与 IPermissionChecker

```csharp
// 方式一：特性标注
[Authorize("Elderly.Read")]
[HttpGet("api/elderly")]
public async Task<PagedResultDto<ElderlyDto>> GetListAsync(
    PagedAndSortedResultRequestDto input) { ... }

// 方式二：代码中手动检查（适用于复杂逻辑）
if (!await _permissionChecker.IsGrantedAsync("Elderly.ReadAll"))

    query = query.Where(e => e.NurseId == CurrentUser.Id);
```

**授权三步走**：定义权限 → 分配给角色 → 在接口上标注。

## 10.9 从请求到授权的完整链路

```
客户端 → Authorization: Bearer ***
  → 1.认证中间件：解析JWT、验证签名、检查过期、提取Claims、设置User
    → 2.授权中间件：检查[Authorize]、验证权限 → 403或继续
      → 3.路由匹配 → Controller → Action → 返回结果
```

```csharp
// 中间件顺序很重要！
app.UseRouting();
app.UseAuthentication();  // 先认证（你是谁）
app.UseAuthorization();   // 再授权（你能做什么）
app.UseEndpoints(endpoints => endpoints.MapControllers());
```

**⚠️ UseAuthentication 必须在 UseAuthorization 之前！认证在前，授权在后。**

## 10.10 微信登录集成原理

### 10.10.1 为什么养老院需要微信登录

老人家属不会记复杂密码，微信是国内最普及的应用。家属扫码登录查看老人状态，体验最好。

### 10.10.2 OAuth2 授权码模式实现

```
家属手机 → 点击微信登录 → 跳转微信授权页 → 扫码确认
  → 微信回调带code → 后端用code换access_token
  → 用access_token获取用户信息 → 查找/创建本地用户
  → 签发系统JWT Token → 返回前端
```

```csharp
public async Task<LoginResultDto> HandleWechatCallbackAsync(string code)
{
    // 1. 用code换微信access_token
    var tokenUrl = $"https://api.weixin.qq.com/sns/oauth2/access_token" +
        $"?appid={_options.AppId}&secret={_options.AppSecret}" +
        $"&code={code}&grant_type=authorization_code";
    var tokenResp = await client.GetStringAsync(tokenUrl);
    var wechatToken = JsonSerializer.Deserialize<WechatTokenResponse>(tokenResp);

    // 2. 获取微信用户信息
    var userInfo = await client.GetStringAsync(
        $"https://api.weixin.qq.com/sns/userinfo" +
        $"?access_token={wechatToken.AccessToken}&openid={wechatToken.OpenId}");

    // 3. 查找或创建本地用户
    var user = await _userManager.FindByLoginAsync("Wechat", wechatToken.OpenId);
    if (user == null)
    {
        user = new IdentityUser(GuidGenerator.Create(), $"wx_{wechatToken.OpenId}",
            $"{wechatUser.Nickname}@wechat.local", CurrentTenant.Id);
        await _userManager.CreateAsync(user);
        await _userManager.AddLoginAsync(user,
            new UserLoginInfo("Wechat", wechatToken.OpenId, "微信"));
        await _userManager.AddToRoleAsync(user, NursingHomeRoles.Family);
    }

    // 4. 签发系统Token
    return await _tokenService.GenerateTokenAsync(user);
}
```

## 实战案例：养老院认证授权系统

**需求**：员工密码登录、家属微信登录、按角色授权、Token无感刷新。

**步骤一**：在 PermissionDefinitionProvider 中定义权限（完整代码见 10.8.1 节）。

**步骤二**：给角色分配权限。

```csharp
// 护工：查看老人 + 读写健康档案
await SetPermissionAsync("Nurse", "Elderly.Read", true);
await SetPermissionAsync("Nurse", "Health.Read", true);
await SetPermissionAsync("Nurse", "Health.Write", true);
// 医生：查看老人 + 读写健康档案
await SetPermissionAsync("Doctor", "Elderly.Read", true);
await SetPermissionAsync("Doctor", "Health.Read", true);
await SetPermissionAsync("Doctor", "Health.Write", true);
// 家属：只读老人基本信息
await SetPermissionAsync("Family", "Elderly.Read", true);
// 财务：读写账单
await SetPermissionAsync("Finance", "Billing.Read", true);
await SetPermissionAsync("Finance", "Billing.Create", true);
await SetPermissionAsync("Finance", "Billing.Confirm", true);
// 管理员：全部权限
await SetPermissionAsync("Admin", "Elderly", true);
await SetPermissionAsync("Admin", "Health", true);
await SetPermissionAsync("Admin", "Billing", true);
```

**步骤三**：Controller 上标注 `[Authorize("PermissionName")]` 保护接口。

```csharp
[Authorize("Elderly.Read")]

[Authorize("Billing.Confirm")]
[HttpPost("api/billing/{id}/confirm")]
public async Task ConfirmPaymentAsync(Guid id)
{
    await _billingAppService.ConfirmPaymentAsync(id);
}
```

**步骤四**：前端用 HttpClient 拦截器实现 401 自动刷新 Token（参见 10.4.2 节代码）。

## 常见错误与最佳实践

| # | 错误写法 | 正确写法 | 为什么错 |
|---|---------|---------|---------|
| 1 | JWT Payload 存放密码、身份证号等敏感信息 | 只放 UserId、UserName、TenantId 等非敏感声明 | Payload 只是 Base64 编码，不是加密，任何人都能解码查看 |
| 2 | 认证中间件放在授权中间件之后 | UseAuthentication 在 UseAuthorization 之前 | 授权依赖认证结果，顺序颠倒导致授权拿不到用户信息 |
| 3 | Refresh Token 存 localStorage | Access Token 存内存变量 + Refresh Token 存 httpOnly Cookie | localStorage 易受 XSS 攻击，httpOnly Cookie 无法被 JS 读取 |
| 4 | Access Token 过期时间设为 7 天 | Access Token 设 30 分钟，Refresh Token 设 7 天 | 长 Token 泄露后攻击窗口大，短 Token + 无感刷新更安全 |
| 5 | 权限定义后忘记在模块中注册 | 在 PermissionDefinitionProvider 中定义并确保模块 DependsOn 正确 | 未注册的权限在运行时静默失效，接口无保护 |
| 6 | 用 AllowAnyOrigin + AllowCredentials 配 CORS | 指定具体域名 WithOrigins + AllowCredentials | ASP.NET Core 直接抛 InvalidOperationException |
| 7 | 微信登录后不校验 state 参数 | 生成随机 state 存 session，回调时校验一致 | 不校验 state 可能遭受 CSRF 攻击，被伪造授权回调 |
| 8 | 所有接口都用同一个权限策略 | 按功能粒度定义权限（查询/新增/修改/删除分别定义） | 粗粒度权限导致"要么全有要么全无"，违反最小权限原则 |

## 本章小结

本章完成了养老院系统的认证授权体系：认证解决身份问题，授权解决权限问题。OAuth2 提供了四种授权模式，JWT 的三段式结构（Header.Payload.Signature）实现无状态认证，Access Token + Refresh Token 实现安全的无感刷新，IdentityServer4 作为统一认证中心，ABP Identity 管理用户角色权限。核心原则：**最小权限——每个角色只授予完成工作所必需的最小权限。**

## 面试题

### 面试题 1（初级 / 概念题）
**题目**：认证和授权有什么区别？

**参考答案**：认证（Authentication）是验证身份——"你是谁"，通过用户名密码、Token、微信扫码等方式确认用户身份。授权（Authorization）是验证权限——"你能做什么"，通过角色和权限策略判断用户是否有权访问某个资源。认证是授权的前提，没有认证就无法授权。养老院场景：护工刷卡进入系统是认证，系统判断护工只能查看自己负责的长者是授权。

### 面试题 2（初级 / 概念题）
**题目**：JWT 由哪几部分组成？

**参考答案**：JWT 由三部分用 `.` 分隔：**Header**（声明算法类型，如 HS256）、**Payload**（存放用户信息，如 UserId、TenantId，Base64Url 编码但不加密）、**Signature**（用密钥对 Header.Payload 签名，防篡改）。格式：`xxxxx.yyyyy.zzzzz`。注意：Payload 只是编码不是加密，任何人都能解码查看，所以不能存放密码等敏感信息。养老院场景：JWT Token 中存放护工 ID 和所属养老院 TenantId。

### 面试题 3（中级 / 概念题）
**题目**：Cookie、Session、Token 三者有什么区别？

**参考答案**：**Cookie**：存储在浏览器，每次请求自动携带，受同源策略限制，容量 4KB。**Session**：存储在服务端（内存/Redis），通过 Cookie 中的 SessionId 关联，有状态，水平扩展需共享存储。**Token（JWT）**：存储在客户端（内存/LocalStorage），无状态，天然支持跨域和移动端，服务端不存储。养老院系统前后端分离架构，用 JWT Token 最合适；如果只有传统 MVC 页面，Cookie + Session 也够用。

### 面试题 4（中级 / 概念题）
**题目**：为什么用 Access Token + Refresh Token 双 Token 机制？

**参考答案**：Access Token 短命（30 分钟），减少泄露后的攻击窗口；Refresh Token 长命（7 天），避免用户频繁登录。当 Access Token 过期时，前端用 Refresh Token 静默换取新 Token，用户无感知。如果 Refresh Token 被盗（同一 Token 在两个地方使用），服务端检测到后立即吊销该用户所有 Token，强制重新登录。养老院场景：护工早上登录后一整天不用重新输入密码。

### 面试题 5（初级 / 概念题）
**题目**：OAuth2 四种授权模式分别适用于什么场景？

**参考答案**：**授权码模式**：第三方登录（微信扫码），最安全，Token 不经过浏览器。**密码模式**：第一方应用（自家 App），用户直接提供用户名密码。**客户端模式**：服务间通信（后台定时任务），无用户参与。**隐式模式**：已废弃，不推荐使用。养老院系统中：家属用微信扫码登录用授权码模式，护工用 App 登录用密码模式，后台定时生成报表用客户端模式。

### 面试题 6（高级 / 场景题）
**题目**：Refresh Token 被重复使用意味着什么？怎么处理？

**参考答案**：Refresh Token 应该一次性使用（用后即废）。如果同一 Token 被使用两次，说明可能被盗——攻击者和合法用户同时持有同一 Token。处理策略：检测到重复使用后，立即吊销该用户的所有 Token（Access + Refresh），强制重新登录，并记录安全日志。养老院场景：护工账号在两台设备同时刷新 Token，系统应立即锁定账号并通知管理员。

### 面试题 7（中级 / 概念题）
**题目**：IdentityServer4 的作用是什么？

**参考答案**：IdentityServer4 是 OAuth2/OpenID Connect 服务器，负责统一认证中心：①签发和验证 JWT Token；②管理客户端（哪些应用可以访问）；③管理 API 资源（哪些接口需要保护）；④支持多种授权模式。养老院系统中，IdentityServer4 作为独立认证服务，所有微服务（长者服务、财务服务、护理服务）都从它获取 Token 并验证。

### 面试题 8（初级 / 概念题）
**题目**：ABP 中如何定义和使用权限？

**参考答案**：三步：①在 `PermissionDefinitionProvider` 的 `Define` 方法中定义权限（`context.AddPermission("Yls.Elder.Create")`）；②通过 `IPermissionManager` 将权限分配给角色（管理后台操作）；③在 Controller 或 AppService 上标注 `[Authorize("Yls.Elder.Create")]`。ABP 还支持 `IPermissionChecker` 在代码中动态检查权限。养老院场景：定义"修改护理等级"权限，只分配给护士长角色。

### 面试题 9（高级 / 概念题）
**题目**：JWT 存在哪里最安全？

**参考答案**：三种方案各有利弊：①**内存变量**：最安全（页面刷新丢失），适合 SPA 单页应用；②**httpOnly Cookie**：防 XSS（JS 无法读取），但需防 CSRF（加 SameSite=Strict）；③**localStorage**：持久化但易受 XSS 攻击。推荐方案：Access Token 存内存变量（短命，刷新丢失可重新获取），Refresh Token 存 httpOnly Cookie（长命，防 XSS）。养老院系统前端是 Vue SPA，适合此方案。

### 面试题 10（初级 / 代码题）
**题目**：认证授权中间件的正确顺序是什么？

**参考答案**：`UseRouting` → `UseAuthentication` → `UseAuthorization` → `UseEndpoints`。认证必须在授权之前，因为授权需要读取认证结果（ClaimsPrincipal）。如果顺序颠倒，授权中间件拿不到用户信息，所有请求都会返回 401。养老院系统 Startup.Configure 中必须按此顺序配置。

### 面试题 11（高级 / 场景题）
**题目**：微信扫码登录的完整流程是什么？

**参考答案**：OAuth2 授权码模式：①前端跳转微信授权页面（带 AppId + RedirectUri + State）；②用户扫码确认；③微信回调 RedirectUri 并带 code + state；④后端校验 state 防 CSRF；⑤用 code 向微信服务器换取 access_token + openid；⑥用 openid 查询是否已绑定养老院系统账号；⑦已绑定则签发 JWT，未绑定则引导绑定手机号。

### 面试题 12（高级 / 设计题）
**题目**：微服务架构如何实现统一认证？

**参考答案**：独立认证中心（IdentityServer4）签发 JWT Token，各微服务用公钥自行验证（无需每次请求都问认证中心）。API 网关在入口处统一验证 Token，转发请求时将用户信息放入 Header。养老院系统中：Ocelot 网关验证 Token → 将 UserId/TenantId 放入 Header → 各微服务从 Header 读取用户信息 → ABP 的 ClaimsPrincipal 自动填充。

### 面试题 13（中级 / 概念题）
**题目**：什么是 CSRF？JWT Token 为什么能防御 CSRF？

**参考答案**：CSRF（跨站请求伪造）利用浏览器自动携带 Cookie 的特性，诱导用户在已登录状态下发起伪造请求。JWT Token 存在内存变量中，需要 JavaScript 手动添加到请求 Header，恶意网站无法获取 Token 也无法自动携带。如果用 Cookie 存 Token 则仍需额外防 CSRF（SameSite + AntiForgeryToken）。养老院系统前后端分离用 JWT 天然防 CSRF。

### 面试题 14（高级 / 场景题）
**题目**：如何为养老院系统实现"只有护士长才能修改护理等级"的权限控制？

**参考答案**：①在 `YlsPermissionDefinitionProvider` 中定义权限 `Yls.NursingLevel.Update`；②在管理后台将此权限分配给"护士长"角色；③在 `NursingLevelAppService` 的 Update 方法上标注 `[Authorize(YlsPermissions.NursingLevel.Update)]`；④护工调用时 ABP 自动返回 403 Forbidden。也可以用 `IPermissionChecker.IsGrantedAsync("Yls.NursingLevel.Update")` 在代码中动态检查。
