## 为什么需要学这个？

养老院系统的业务规则复杂且涉及金钱。账单计算、护理费、床位费——任何环节出错都直接影响长者和家属的切身利益。

> **真实场景**：某养老院系统上线后，开发人员修改了账单计算逻辑，将护理费从"按天计算"改成了"按月计算"。由于没有单元测试，上线后才发现已多收了 200 多位长者家属的钱。家属投诉、财务退款、管理层问责，团队花了两周善后。

如果当时有一个测试：

```csharp
[Fact]
public void 护理费应按天计算()
{
    var result = _billingService.CalculateNursingFee(dailyRate: 100m, days: 30);
    Assert.Equal(3000m, result);
}
```

这个 bug 就会在代码提交时被发现，而不是上线后被家属投诉时才暴露。

**测试不是负担，而是保险。** 对于养老院系统来说，测试更是对长者和家属的一种责任。

## 1. xUnit 基础

xUnit 是 .NET 最流行的测试框架，也是 ABP 官方推荐框架。

### 1.1 创建测试项目

```bash
dotnet new xunit -n Yls.ElderlyCare.Tests
dotnet add Yls.ElderlyCare.Tests reference Yls.ElderlyCare.Application
```

### 1.2 Fact 与 Theory

**Fact** —— 无参数测试，执行一次：

```csharp
[Fact]
public void 床位费计算_按天数计算()
{
    var dailyRate = 150m;
    var days = 30;
    var total = dailyRate * days;
    Assert.Equal(4500m, total);
}
```

**Theory** —— 参数化测试，多组输入验证同一逻辑：

```csharp
[Theory]
[InlineData(CareLevel.Basic, 100)]
[InlineData(CareLevel.Standard, 200)]
[InlineData(CareLevel.Intensive, 350)]
public void 各护理等级对应正确日费率(CareLevel level, decimal expectedRate)
{
    var rate = CareLevelRateConfig.GetDailyRate(level);
    Assert.Equal(expectedRate, rate);
}
```

### 1.3 MemberData

参数较复杂时使用 MemberData：

```csharp
public static IEnumerable<object[]> 月度账单测试数据()
{
    yield return new object[] { 150m, 200m, 30, 10500m };
    yield return new object[] { 200m, 350m, 28, 15400m };
}

[Theory]
[MemberData(nameof(月度账单测试数据))]
public void 月度账单应正确汇总(decimal bedRate, decimal nursingRate, int days, decimal expected)
{
    var total = BillingCalculator.Calculate(bedRate, nursingRate, days);
    Assert.Equal(expected, total);
}
```

### 1.4 Assert 断言方法

```csharp
Assert.Equal(expected, actual);           // 相等
Assert.NotEqual(expected, actual);        // 不相等
Assert.Null(result);                      // 空
Assert.NotNull(result);                   // 非空
Assert.True(condition);                   // 为真
Assert.False(condition);                  // 为假
Assert.Contains(item, collection);        // 包含
Assert.InRange(actual, low, high);        // 范围
Assert.Throws<BusinessException>(() =>);  // 异常
```

## 3. NSubstitute Mock 依赖

在单元测试中，我们只关心被测代码的逻辑，不关心它依赖的外部系统。NSubstitute 创建"替身"对象。

### 3.1 基本用法

```csharp
public class BillingAppServiceTests
{
    private readonly IRepository<Elder, Guid> _elderRepo;
    private readonly IBillingCalculator _calculator;
    private readonly BillingAppService _service;

    public BillingAppServiceTests()
    {
        _elderRepo = Substitute.For<IRepository<Elder, Guid>>();
        _calculator = Substitute.For<IBillingCalculator>();
        _service = new BillingAppService(_elderRepo, _calculator);
    }

    [Fact]
    public async Task 长者不存在时_应抛出异常()
    {
        _elderRepo.GetAsync(Arg.Any<Guid>()).Returns((Elder)null);
        await Assert.ThrowsAsync<EntityNotFoundException>(
            () => _service.GenerateBillAsync(Guid.NewGuid()));
    }

    [Fact]
    public async Task 生成账单_应调用计算器()
    {
        var elder = new Elder { Id = Guid.NewGuid(), Name = "张大爷" };
        _elderRepo.GetAsync(elder.Id).Returns(elder);
        _calculator.Calculate(Arg.Any<BillingInput>()).Returns(new BillResult { Total = 5000m });

        var result = await _service.GenerateBillAsync(elder.Id);

        Assert.Equal(5000m, result.Total);
        _calculator.Received(1).Calculate(Arg.Any<BillingInput>());
    }
}
```

### 3.2 常用 API

```csharp
_repo.GetAsync(id).Returns(elder);                          // 设置返回值
_repo.GetAsync(Arg.Any<Guid>());                            // 任意参数
_repo.Received(1).DeleteAsync(Arg.Any<Guid>());             // 验证调用次数
_repo.DidNotReceive().UpdateAsync(Arg.Any<Elder>());        // 验证未调用
```

## 4. 测试分层策略

养老院系统测试分三层，从底层到顶层逐层验证：

**仓储测试（集成测试）** —— 使用真实数据库或内存数据库，验证数据访问：

```csharp
public class ElderRepositoryTests : AbpIntegratedTest<ElderlyCareTestModule>
{
    private readonly IRepository<Elder, Guid> _elderRepo;
    private readonly IUnitOfWorkManager _unitOfWorkManager;

    public ElderRepositoryTests()
    {
        _elderRepo = GetRequiredService<IRepository<Elder, Guid>>();
        _unitOfWorkManager = GetRequiredService<IUnitOfWorkManager>();
    }

    [Fact]
    public async Task 插入长者_应能正确查询()
    {
        using var uow = _unitOfWorkManager.Begin();
        var elder = new Elder { Id = Guid.NewGuid(), Name = "李奶奶", Age = 78 };
        await _elderRepo.InsertAsync(elder);
        await uow.CompleteAsync();

        using var uow2 = _unitOfWorkManager.Begin();
        var found = await _elderRepo.GetAsync(elder.Id);
        Assert.Equal("李奶奶", found.Name);
    }
}
```

**服务测试（单元测试）** —— Mock 仓储，只测试业务逻辑。最常见的测试类型。

**API 测试（端到端）** —— 通过 HTTP 调用验证完整链路：

```csharp
public class ElderController_Tests : AbpAspNetCoreIntegratedTestBase<Program>
{
    [Fact]
    public async Task Get_Elders_Should_Return_Success()
    {
        var response = await Client.GetAsync("/api/app/elder");
        response.EnsureSuccessStatusCode();
        var content = await response.Content.ReadAsStringAsync();
        Assert.NotNull(content);
    }
}
```

> **总结**：仓储测试验证数据访问，服务测试验证业务逻辑，API 测试验证整体流程。三层互补。

## 5. Shouldly 断言库

Shouldly 让断言读起来像自然语言：

```csharp
// 原生 Assert
Assert.Equal(78, elder.Age);
Assert.True(elder.Age >= 60);
Assert.Contains("张", elder.Name);

// Shouldly —— 读起来像一句话
elder.Age.ShouldBe(78);
elder.Age.ShouldBeGreaterThanOrEqualTo(60);
elder.Name.ShouldContain("张");
```

集合断言和异常断言：

```csharp
[Fact]
public void 查询结果_应包含所有住院长者()
{
    var elders = _service.GetActiveElders();
    elders.ShouldNotBeEmpty();
    elders.ShouldContain(e => e.Name == "张大爷");
    elders.ShouldAllBe(e => e.Status == AdmissionStatus.Active);
}

[Fact]
public void 费率为负数_应抛出异常()
{
    Should.Throw<BusinessException>(() =>
        _calculator.CalculateNursingFee(dailyRate: -100m, days: 30));
}
```

> **提示**：Shouldly 和原生 Assert 可以混用，建议新代码统一使用 Shouldly。

## 6. 测试覆盖率

**测试覆盖率** = 被测试执行到的代码行数 / 总代码行数 × 100%

**不追求 100%**。100% 覆盖率不代表没有 Bug，只代表每行代码都被执行过。

| 代码类型 | 建议覆盖率 | 原因 |
|---------|-----------|------|
| 核心业务逻辑 | 80%+ | 直接影响金钱和长者权益 |
| 应用服务层 | 60-80% | 包含重要编排逻辑 |
| 控制器/API 层 | 30-50% | 很多是框架生成代码 |

```bash
dotnet test /p:CollectCoverage=true /p:CoverletOutputFormat=cobertura
```

> **养老院建议**：优先保证账单计算、入住退住、护理等级变更这三个核心流程的覆盖。

## 7. 实战：为长者入住流程编写完整测试

### 7.1 业务规则

1. 年龄必须 ≥ 60 岁
2. 床位必须是"空闲"状态
3. 同一长者不能重复入住
4. 入住成功后床位状态变为"已占用"
5. 入住成功后创建入住记录

### 7.2 服务层测试

```csharp
public class AdmissionAppServiceTests
{
    private readonly IRepository<Elder, Guid> _elderRepo;
    private readonly IRepository<Bed, Guid> _bedRepo;
    private readonly IRepository<AdmissionRecord, Guid> _recordRepo;
    private readonly AdmissionAppService _service;

    public AdmissionAppServiceTests()
    {
        _elderRepo = Substitute.For<IRepository<Elder, Guid>>();
        _bedRepo = Substitute.For<IRepository<Bed, Guid>>();
        _recordRepo = Substitute.For<IRepository<AdmissionRecord, Guid>>();
        _service = new AdmissionAppService(_elderRepo, _bedRepo, _recordRepo);
    }

    [Theory]
    [InlineData(55)]
    [InlineData(59)]
    public async Task 年龄不足60岁_入住应被拒绝(int age)
    {
        var dto = new CheckInDto { ElderName = "张三", Age = age, BedId = Guid.NewGuid() };
        var ex = await Should.ThrowAsync<BusinessException>(
            () => _service.CheckInAsync(dto));
        ex.Message.ShouldContain("年龄");
    }

    [Fact]
    public async Task 床位已被占用_入住应被拒绝()
    {
        var bedId = Guid.NewGuid();
        _bedRepo.GetAsync(bedId).Returns(new Bed { Id = bedId, Status = BedStatus.Occupied });
        var dto = new CheckInDto { ElderName = "王奶奶", Age = 75, BedId = bedId };
        var ex = await Should.ThrowAsync<BusinessException>(
            () => _service.CheckInAsync(dto));
        ex.Message.ShouldContain("床位");
    }

    [Fact]
    public async Task 满足所有条件_应成功入住()
    {
        var bedId = Guid.NewGuid();
        _bedRepo.GetAsync(bedId).Returns(new Bed { Id = bedId, Status = BedStatus.Available });
        var dto = new CheckInDto { ElderName = "李大爷", Age = 72, BedId = bedId, CareLevel = CareLevel.Standard };

        var result = await _service.CheckInAsync(dto);

        result.ShouldNotBeNull();
        result.ElderName.ShouldBe("李大爷");
        result.BedId.ShouldBe(bedId);
        await _bedRepo.Received(1).UpdateAsync(Arg.Is<Bed>(b => b.Status == BedStatus.Occupied));
        await _recordRepo.Received(1).InsertAsync(Arg.Any<AdmissionRecord>());
    }

    [Fact]
    public async Task 已在住的长者_不能再次入住()
    {
        var elderId = Guid.NewGuid();
        _elderRepo.GetAsync(elderId).Returns(new Elder { Id = elderId, Status = AdmissionStatus.Active });
        var dto = new CheckInDto { ElderId = elderId, BedId = Guid.NewGuid() };
        var ex = await Should.ThrowAsync<BusinessException>(
            () => _service.CheckInAsync(dto));
        ex.Message.ShouldContain("已入住");
    }
}
```

### 7.3 测试命名建议

| 风格 | 示例 |
|------|------|
| 中文场景（推荐） | `年龄不足60岁_入住应被拒绝` |
| 英文下划线 | `CheckIn_AgeBelow60_ShouldBeRejected` |

养老院项目面向国内团队，推荐中文命名，方便业务人员理解测试意图。

## 常见错误与最佳实践

| # | 错误写法 | 正确写法 | 为什么错 |
|---|---------|---------|---------|
| 1 | 一个测试方法测多个场景 | 每个测试只验证一个场景 | 第一个断言失败后面的不执行，无法精确定位 |
| 2 | 测试连接生产数据库 | 使用内存数据库或独立测试库 | 测试数据会污染生产环境，可能误删真实长者数据 |
| 3 | Mock 所有依赖包括值对象 | 只 Mock 外部依赖（仓储、第三方服务） | 值对象无副作用，Mock 会掩盖真实计算问题 |
| 4 | 测试名用 `Test1`、`Test2` | 用业务场景命名如 `床位被占用时_入住应失败` | 模糊名称无法从报告判断哪个场景失败 |
| 5 | 测试依赖上一个测试的状态 | 每个测试独立准备数据 | 测试执行顺序不确定，间歇性失败 |
| 6 | 为凑覆盖率写无意义测试 | 只测有业务价值的代码 | 无意义测试增加维护成本，不带来质量保障 |
| 7 | 测试中用 `Thread.Sleep` 等待异步 | 使用 `await` 正确等待 | Sleep 时间不确定，太快失败太慢浪费 |
| 8 | 测试代码写在生产项目中 | 单独创建测试项目 | 测试依赖不应打包到生产部署中 |

## 面试题

### 面试题 1（初级 / 概念题）
**题目**：什么是单元测试？为什么养老院系统需要单元测试？

**参考答案**：单元测试是针对代码中最小可测试单元进行验证的自动化测试。养老院系统的账单计算、护理等级变更等业务直接涉及金钱，一旦出错影响重大。单元测试能在代码提交阶段就发现 bug，避免账单计算逻辑改错后上线才发现问题。它相当于给代码买了一份保险，用少量编写时间换取大量排查时间。

### 面试题 2（初级 / 代码题）
**题目**：xUnit 中 `[Fact]` 和 `[Theory]` 有什么区别？

**参考答案**：`[Fact]` 标记无参数测试，执行一次。`[Theory]` 标记参数化测试，配合 `[InlineData]` 或 `[MemberData]` 用多组参数执行多次。养老院系统中验证不同护理等级的费率时，用 `[Theory]` + `[InlineData]` 可以一次性测试所有等级，减少重复代码。

### 面试题 3（中级 / 概念题）
**题目**：什么是 Mock？为什么要 Mock 依赖？

**参考答案**：Mock 是创建替身对象来替代真实依赖。比如 `BillingAppService` 依赖 `IRepository<Elder>` 查询长者信息，测试时用 NSubstitute 创建 Mock 仓储，手动设置返回数据。这样测试只验证业务逻辑本身，不受数据库状态影响。例如 Mock 返回费率 200 元/天的长者，验证 30 天账单是否为 6000 元。

### 面试题 4（中级 / 代码题）
**题目**：如何用 NSubstitute 验证某个方法被调用了指定次数？

**参考答案**：使用 `Received()` 方法。养老院入住测试中验证床位状态更新：`await _bedRepo.Received(1).UpdateAsync(Arg.Is<Bed>(b => b.Status == BedStatus.Occupied))`，表示 UpdateAsync 应被调用恰好 1 次。验证未被调用用 `_repo.DidNotReceive()`。

### 面试题 5（中级 / 概念题）
**题目**：仓储测试、服务测试、API 测试分别验证什么？

**参考答案**：仓储测试验证数据访问层操作是否正确，优点是接近真实场景，缺点依赖数据库较慢。服务测试用 Mock 替代仓储验证业务逻辑，优点是快且聚焦，缺点不验证真实数据库交互。API 测试通过 HTTP 调用验证完整链路，优点覆盖最全，缺点最慢且维护成本高。养老院项目建议以服务测试为主。

### 面试题 6（高级 / 概念题）
**题目**：什么是测试覆盖率？为什么不应追求 100%？

**参考答案**：测试覆盖率是被测试执行到的代码行数占总代码行数的比例。100% 只代表每行代码被执行过，不代表逻辑正确。养老院系统应优先保证账单计算、入住退住等核心流程的高覆盖率，DTO 和枚举等无逻辑代码不需追求覆盖。过度追求覆盖率会导致团队编写无意义测试，增加维护成本。

### 面试题 7（初级 / 代码题）
**题目**：Shouldly 和原生 Assert 有什么区别？

**参考答案**：原生写法 `Assert.Equal(78, elder.Age)`，Shouldly 写法 `elder.Age.ShouldBe(78)`。Shouldly 语法更自然，读起来像"年龄应该是78岁"，断言失败时也提供更好的错误信息。养老院项目中使用 Shouldly 让非技术人员也能理解测试意图，如 `result.Total.ShouldBeGreaterThanOrEqualTo(0)`。

### 面试题 8（高级 / 设计题）
**题目**：养老院系统中哪些模块应优先编写测试？

**参考答案**：第一优先账单计算模块，涉及金钱出错后果最严重；第二优先入住退住流程，涉及床位和长者状态联动；第三优先护理等级变更，影响后续费用计算。测试策略上核心业务用服务层单元测试覆盖所有分支，重要端到端流程用集成测试补充，UI 和 API 层做冒烟测试即可。聚焦高风险区域而非追求全覆盖。

### 面试题 9（中级 / 概念题）
**题目**：集成测试和单元测试的区别是什么？

**参考答案**：单元测试只测试一个类或方法的逻辑，所有外部依赖被 Mock，速度快但不验证真实交互。集成测试启动真实依赖（如内存数据库），验证多个组件协作，较慢但更接近真实。养老院系统验证"账单=床位费+护理费"用单元测试即可；验证"入住后数据库能查到记录"需要集成测试。两者互补。

### 面试题 10（高级 / 设计题）
**题目**：测试代码应该和生产代码放在同一个项目中吗？

**参考答案**：不应该。测试代码应放在独立测试项目中。第一，测试依赖（xUnit、NSubstitute）不应打包到生产部署中；第二，测试项目可引用生产项目但反之不行，保持依赖方向单一；第三，独立项目可单独编译和运行，方便 CI/CD 集成。养老院系统的生产环境只应包含业务代码，Fake 实现和测试数据不应出现在部署包中。

### 面试题 11（高级 / 代码题）
**题目**：如何测试一个会抛出异常的业务方法？

**参考答案**：xUnit 使用 `Assert.ThrowsAsync<BusinessException>(() => service.MethodAsync())`。Shouldly 使用 `Should.ThrowAsync<BusinessException>(() => service.MethodAsync())`。养老院场景中年龄不足 60 岁办理入住，应断言抛出 BusinessException 且消息包含"年龄"。注意不仅验证异常类型，还要验证异常消息是否正确。

### 面试题 12（高级 / 代码题）
**题目**：ABP 框架中如何配置集成测试的测试模块？

**参考答案**：创建类继承 AbpModule 并标记 `[DependsOn]` 依赖主应用模块和 EF Core 模块。在 `ConfigureServices` 中用 `UseInMemoryDatabase("TestDb")` 替换数据库，用 `context.Services.Replace()` 替换外部依赖为 Fake 实现。测试类继承 `AbpIntegratedTest<YourTestModule>`，通过 `GetRequiredService<T>()` 获取服务实例。会启动完整 ABP 容器包括依赖注入和工作单元。
