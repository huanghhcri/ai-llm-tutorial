# 第 11 章：多租户架构

## 学习目标

- 理解多租户的核心概念及在养老院管理系统中的实际意义

- 学会配置 ABP 的四种租户解析策略
- 熟练使用 `IMultiTenant` 接口实现数据自动过滤
- 掌握租户切换、租户级设置与功能开关的用法

## 前置知识

- 已完成第 04 章学习（ABP 框架深度解析：模块系统、DDD 分层、AutoMapper、工作单元）

## 为什么需要学这个？

你开发了一套养老院管理系统，最初只服务一家养老院。突然投资人说："我们要在全省开二十家连锁养老院，每家都要用这套系统。"没有多租户架构，你就要部署二十套独立系统，修一个 Bug 改二十次——这是运维噩梦。

多租户的核心思想：**一套代码、一个部署、多个租户各自隔离**。就像一栋公寓楼，每户独立门锁，共用电梯和物业管理。养老院 A 看不到养老院 B 的数据，但所有养老院用同一套系统，统一升级维护。

## 11.1 什么是多租户

**多租户（Multi-Tenancy）** 指同一应用实例同时为多个独立组织提供服务，确保数据和配置相互隔离。在养老院系统中，每个养老院就是一个租户：阳光养老院、幸福养老院、康乐养老院共享同一套程序，但各自拥有独立的老人档案和财务数据。

生活类比——公寓楼模型：

| 公寓楼概念 | 多租户对应 | 养老院场景 |
|-----------|-----------|-----------|
| 公寓楼 | 应用实例 | 颐养天年平台 |
| 各户房间 | 租户数据空间 | 各养老院的数据 |
| 门锁钥匙 | 认证授权 | 租户识别与权限 |
| 公共电梯 | 共享基础设施 | 通用服务模块 |
| 物业管理 | 宿主管理 | 平台管理员 |

## 11.2 三种数据隔离方案对比

**方案一：共享数据库 + TenantId 列**——所有租户数据在同一张表中，通过 TenantId 区分。ABP 默认方案。

**方案二：独立 Schema**——同一数据库中为每个租户创建独立 Schema，各有一份数据表。

**方案三：独立数据库**——每个租户拥有完全独立的数据库实例，物理级别隔离。

| 对比维度 | 共享数据库+TenantId | 独立Schema | 独立数据库 |
|---------|-------------------|-----------|-----------|
| **部署成本** | 最低 | 中等 | 最高 |
| **数据隔离性** | 逻辑隔离 | Schema级隔离 | 物理隔离 |
| **实现复杂度** | 最简单 | 需路由逻辑 | 需连接管理 |
| **查询性能** | 需索引优化 | 有切换开销 | 最优 |
| **扩展上限** | 数千租户 | 数百租户 | 数十租户 |
| **适合场景** | SaaS平台 | 中等安全 | 金融级安全 |

养老院系统通常几十到几百家租户，数据量适中，**方案一完全胜任**。

## 11.3 ABP 租户解析策略

当用户访问系统时，ABP 需要知道"你是哪个养老院的"。四种解析策略：

**域名解析**——根据子域名判断，适合每个养老院有独立子域名的场景：
```
sunshine.nursinghome.com → 阳光养老院
happiness.nursinghome.com → 幸福养老院
```

**Header 解析**——通过 `X-TenantId` 请求头传递，适合 API 调用和移动端。

**Cookie 解析**——通过 Cookie 传递，适合 Web 浏览器场景。

**QueryString 解析**——通过 URL 参数 `?tenantId=xxx` 传递，适合开发调试。

配置示例（按优先级从高到低）：

```csharp
Configure<AbpTenantResolveOptions>(options =>
{
    // 域名解析（生产环境主策略）
    options.TenantResolvers.Add(
        new DomainTenantResolver("{0}.nursinghome.com"));
    // Header 解析（API调用辅助）
    options.TenantResolvers.Add(
        new HeaderTenantResolveContributor("X-TenantId"));
    // Cookie 解析（Web 浏览器场景，登录后写入租户 Cookie）
    options.TenantResolvers.Add(
        new CookieTenantResolveContributor("Abp.TenantId"));
    // QueryString 解析（开发调试用）
    options.TenantResolvers.Add(
        new QueryStringTenantResolveContributor("tenantId"));
});
```

ABP 按添加顺序依次尝试，**第一个成功的结果生效**。

## 11.4 IMultiTenant 接口

实体只需实现 `IMultiTenant` 接口，框架自动为所有查询添加 `WHERE TenantId` 过滤：

```csharp
using System;
using Volo.Abp.Domain.Entities;

namespace NursingHome.Elderly
{
    public class ElderlyPerson : Entity<Guid>, IMultiTenant
    {
        public Guid? TenantId { get; set; }
        public string Name { get; set; }
        public int Age { get; set; }
        public string RoomNumber { get; set; }

        protected ElderlyPerson() { }

        public ElderlyPerson(Guid id, Guid? tenantId, string name, int age)
            : base(id)
        {
            TenantId = tenantId;
            Name = name;
            Age = age;
        }
    }
}
```

```csharp
public class NursingRecord : Entity<Guid>, IMultiTenant
{
    public Guid? TenantId { get; set; }
    public Guid ElderlyPersonId { get; set; }
    public string Content { get; set; }
    public string NurseName { get; set; }

    protected NursingRecord() { }

    public NursingRecord(Guid id, Guid? tenantId, Guid elderlyPersonId,
        string content, string nurseName) : base(id)
    {
        TenantId = tenantId;
        ElderlyPersonId = elderlyPersonId;
        Content = content;
        NurseName = nurseName;
    }
}
```

自动过滤效果：

```csharp
// 你写的代码
var list = await _elderlyRepository.GetListAsync();

// EF Core 实际执行的 SQL
// SELECT * FROM ElderlyPerson WHERE TenantId = '当前租户GUID'
```

即使用 LINQ 复合查询，过滤也会自动叠加：
```csharp
var seniors = await _elderlyRepository
    .Where(e => e.Age >= 80).ToListAsync();
// 实际: WHERE TenantId = @xxx AND Age >= 80
```

## 11.5 租户切换与数据过滤原理

**ICurrentTenant** 接口管理当前请求的租户上下文：

```csharp
var tenantId = _currentTenant.Id;    // 当前租户ID
var isHost = !_currentTenant.Id.HasValue;  // 是否为宿主
```

**临时切换**——宿主管理员跨租户统计时使用 `Change` 方法：

```csharp
public async Task<Dictionary<string, int>> GetCountByTenantAsync(
    List<Guid> tenantIds)
{
    var result = new Dictionary<string, int>();
    foreach (var tenantId in tenantIds)
    {
        using (_currentTenant.Change(tenantId))
        {
            // 此作用域内，所有查询自动过滤为该租户数据
            var count = await _elderlyRepository.GetCountAsync();
            result.Add(tenantId.ToString(), count);
        }
        // 离开 using 后，上下文自动恢复
    }
    return result;
}
```

**与第 5 章 DataFilter 的关系**：多租户过滤本质上也是 EF Core 全局查询过滤器，与软删除过滤（`ISoftDelete`）同属一套体系：

```
ABP 数据过滤器体系
├── ISoftDelete  → WHERE IsDeleted = 0
├── IMultiTenant → WHERE TenantId = @currentTenantId
└── 自定义过滤器  → 业务逻辑过滤
```

三个过滤器可同时生效，查询条件自动叠加。

## 11.6 租户级别的设置和功能开关

不同养老院可以有不同的收费标准和功能模块。通过 ABP 设置系统实现：

```csharp
public class NursingHomeSettingProvider : SettingDefinitionProvider
{
    public override void Define(ISettingDefinitionContext context)
    {
        context.Add(new SettingDefinition(
            "NursingHome.MonthlyFee", "5000",
            isVisibleToClients: true, isInherited: false));

        context.Add(new SettingDefinition(
            "NursingHome.EnableRemoteVisiting", "false",
            isVisibleToClients: true, isInherited: false));
    }
}
```

为不同租户设置不同值（通过种子数据或管理界面），然后在业务代码中读取：

```csharp
public async Task<decimal> GetMonthlyFeeAsync()
{
    // 阳光养老院调用返回5000，幸福养老院调用返回8000
    string fee = await _settingProvider.GetOrNullAsync("NursingHome.MonthlyFee");
    return decimal.Parse(fee ?? "5000");
}
```

`ISettingProvider` 内部自动使用 `ICurrentTenant.Id` 作为 key 查询，同一方法在不同租户上下文中返回不同结果。

**功能开关**：控制某个养老院是否能使用特定功能模块（如智能监护、膳食管理）。与设置管理的区别是：设置管理控制"功能内部参数值"，功能开关控制"功能模块是否可用"。

```csharp
// 步骤一：定义功能
public class NursingHomeFeatureProvider : FeatureDefinitionProvider
{
    public override void Define(IFeatureDefinitionContext context)
    {
        var group = context.AddGroup("NursingHome", "养老院功能");
        group.AddFeature(
            "NursingHome.SmartMonitoring",       // 功能标识
            "false",                              // 默认值：关闭
            displayName: "智能监护模块",
            description: "是否启用智能监护功能（体征实时监测、异常告警）");
        group.AddFeature(
            "NursingHome.MealManagement",        // 功能标识
            "true",                               // 默认值：开启
            displayName: "膳食管理模块",
            description: "是否启用膳食管理功能（营养配餐、过敏源管理）");
    }
}

// 步骤二：在种子数据中为不同租户启用不同功能
// 阳光养老院（高级版）：启用智能监护
await _featureManager.SetForTenantAsync(sunshineTenantId,
    "NursingHome.SmartMonitoring", "true");

// 步骤三：在业务代码中检查功能是否启用
public class SmartMonitoringAppService : ApplicationService
{
    private readonly IFeatureChecker _featureChecker;

    public SmartMonitoringAppService(IFeatureChecker featureChecker)
    {
        _featureChecker = featureChecker;
    }

    public async Task<MonitoringDashboardDto> GetDashboardAsync()
    {
        // 检查当前租户是否启用了智能监护
        if (!await _featureChecker.IsEnabledAsync("NursingHome.SmartMonitoring"))
        {
            throw new BusinessException("SmartMonitoring.NotEnabled",
                "您的养老院未启用智能监护功能，请联系管理员开通。");
        }
        // ... 正常业务逻辑
    }
}

// 步骤四：也可以用特性标注在 Controller 级别拦截
[RequiresFeature("NursingHome.SmartMonitoring")]
public class SmartMonitoringController : AbpController { ... }
```

> 💡 功能开关与设置管理的对比：功能开关控制「有没有」（是否启用某模块），设置管理控制「是多少」（模块内部参数值）。两者可配合使用——先检查功能是否启用，再读取功能内部配置。

## 11.7 实战：为养老院系统添加多租户支持

**步骤一：配置租户解析**

```csharp
[DependsOn(typeof(AbpAspNetCoreMultiTenancyModule))]
public class NursingHomeHttpApiHostModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpTenantResolveOptions>(options =>
        {
            options.TenantResolvers.Add(
                new DomainTenantResolver("{0}.nursinghome.com"));
            options.TenantResolvers.Add(
                new HeaderTenantResolveContributor("X-TenantId"));
            options.TenantResolvers.Add(
                new QueryStringTenantResolveContributor("tenantId"));
        });
    }
}
```

**步骤二：实体标记 IMultiTenant**（见 11.4 节代码）

**步骤三：种子数据**

```csharp
public class NursingHomeDataSeeder : IDataSeedContributor, ITransientDependency
{
    private readonly ITenantRepository _tenantRepository;
    private readonly IRepository<ElderlyPerson, Guid> _elderlyRepository;
    private readonly ICurrentTenant _currentTenant;

    public NursingHomeDataSeeder(
        ITenantRepository tenantRepository,
        IRepository<ElderlyPerson, Guid> elderlyRepository,
        ICurrentTenant currentTenant)
    {
        _tenantRepository = tenantRepository;
        _elderlyRepository = elderlyRepository;
        _currentTenant = currentTenant;
    }

    public async Task SeedAsync(DataSeedContext context)
    {
        var sunshine = await _tenantRepository.FindByNameAsync("sunshine");
        if (sunshine == null)
        {
            sunshine = new Tenant(GuidGenerator.Create(), "sunshine", "阳光养老院");
            await _tenantRepository.InsertAsync(sunshine);
        }

        using (_currentTenant.Change(sunshine.Id))
        {
            if (await _elderlyRepository.GetCountAsync() == 0)
            {
                await _elderlyRepository.InsertAsync(
                    new ElderlyPerson(GuidGenerator.Create(), sunshine.Id, "张大爷", 82));
                await _elderlyRepository.InsertAsync(
                    new ElderlyPerson(GuidGenerator.Create(), sunshine.Id, "李奶奶", 78));
            }
        }
    }
}
```

**步骤四：测试验证**

```csharp
[Fact]
public async Task Should_Filter_Elderly_By_Tenant()
{
    var sunshineId = TestData.SunshineTenantId;
    var happinessId = TestData.HappinessTenantId;

    using (_currentTenant.Change(sunshineId))
    {
        var list = await _elderlyRepository.GetListAsync();
        list.ShouldNotBeEmpty();

    }

    using (_currentTenant.Change(happinessId))
    {
        var list = await _elderlyRepository.GetListAsync();
        list.ShouldNotBeEmpty();
        list.ForEach(p => p.TenantId.ShouldBe(happinessId));
    }
}
```

## 常见错误与最佳实践

| # | 错误写法 | 正确写法 | 为什么错 |
|---|---------|---------|---------|
| 1 | 手动写 `.Where(e => e.TenantId == id)` | 实现 `IMultiTenant` 让框架自动过滤 | 手写遗漏会导致数据泄漏 |
| 2 | 种子数据不切换租户直接插入 | 用 `using (_currentTenant.Change(tenantId))` 包裹 | TenantId 为空或错误 |
| 3 | 所有实体都标记 `IMultiTenant` | 仅业务数据实体标记，全局共享实体不标记 | 共享字典表被过滤导致冗余 |
| 4 | `Change` 不用 `using` 包裹 | 始终用 `using` 确保上下文自动恢复 | 后续代码在错误租户中执行 |
| 5 | 生产环境只配 QueryString 解析 | 域名或 Header 为主，QueryString 仅调试 | URL 暴露租户ID有安全风险 |
| 6 | 用 `if(tenantId==xxx)` 硬编码配置 | 使用 `ISettingProvider` 读取 | 新增租户必须改代码重新部署 |
| 7 | 宿主和租户请求混用不区分 | 明确区分 `TenantId=null` 和有值 | 操作写入错误租户的数据 |
| 8 | 一个 `using` 块里做大量操作 | 切换后尽快完成操作并释放 | 长时间占用错误上下文有隐患 |

## 本章小结

本章从概念到实战讲解了 ABP 多租户架构。核心要点：多租户的本质是一套代码服务多个租户、数据逻辑隔离；共享数据库+TenantId 是最常用隔离方案；四种解析策略按优先级组合使用；`IMultiTenant` 接口从机制上杜绝数据泄漏；`ICurrentTenant.Change` 支持宿主跨租户操作；租户级设置通过 `ISettingProvider` 实现个性化配置。多租户过滤与软删除过滤一样，都是 EF Core 全局查询过滤器，可同时生效。

## 面试题

### 面试题 1（初级 / 概念题）
**题目**：什么是多租户架构？请结合养老院管理系统举例说明。

**参考答案**：多租户架构是一个应用实例同时为多个独立租户提供服务的架构模式。在养老院系统中，阳光养老院和幸福养老院各是一个租户，它们共享同一套代码和基础设施，但各自的数据相互隔离。就像公寓楼每户独立门锁但共享电梯。好处是降低运维成本、统一升级，同时保证数据安全。

### 面试题 2（初级 / 概念题）
**题目**：请对比三种数据隔离方案，并说明养老院系统应选哪种。

**参考答案**：三种方案是共享数据库加 TenantId、独立 Schema 和独立数据库。共享数据库成本最低实现最简单；独立 Schema 隔离性稍好但增加路由复杂度；独立数据库隔离性最强但运维成本成倍增长。养老院系统通常几十到几百家租户，数据量适中，选共享数据库加 TenantId 方案最合适。

### 面试题 3（中级 / 概念题）
**题目**：ABP 如何实现 `IMultiTenant` 的自动数据过滤？说明底层机制。

**参考答案**：ABP 通过 EF Core 的全局查询过滤器机制实现。实体实现 `IMultiTenant` 后，ABP 在 `OnModelCreating` 中自动注册 `WHERE TenantId = @currentTenantId` 过滤条件，每次查询自动附加。当前租户 ID 通过 `ICurrentTenant` 在请求上下文中获取，确保阳光养老院查询只返回阳光的数据，不会误查幸福养老院。

### 面试题 4（中级 / 概念题）
**题目**：ABP 有哪几种租户解析策略？生产环境如何配置？

**参考答案**：有域名解析、Header 解析、Cookie 解析和 QueryString 解析四种。生产环境建议域名解析为主（如 `sunshine.nursinghome.com`），Header 解析为辅（移动端传 `X-TenantId`），QueryString 仅作调试用。按优先级配置后 ABP 依次尝试，第一个成功的结果生效。

### 面试题 5（中级 / 场景题）
**题目**：宿主管理员需要统计所有养老院入住人数，如何实现？

**参考答案**：遍历所有租户，使用 `ICurrentTenant.Change(tenantId)` 临时切换上下文，然后调用仓储的 `GetCountAsync`，该方法自动按当前租户过滤。必须用 `using` 语句包裹确保每次循环结束后上下文恢复，避免后续操作在错误租户中执行。最终汇总各养老院数据返回。

### 面试题 6（高级 / 设计题）
**题目**：全局共享数据（如省份字典表）和租户数据如何区分？

**参考答案**：全局共享数据不实现 `IMultiTenant`，所有租户共用同一份数据，不会被过滤。业务数据（老人、护理记录）必须实现 `IMultiTenant`，每个养老院独立维护。判断标准是该数据是否需要每个养老院独立。种子数据时，全局数据在宿主上下文插入，租户数据在对应租户上下文插入。

### 面试题 7（高级 / 概念题）
**题目**：`ICurrentTenant.Change` 的工作原理是什么？为什么必须用 `using`？

**参考答案**：`Change` 利用 `AsyncLocal<T>` 存储租户上下文，调用时保存当前 ID 并设置新 ID，返回 `IDisposable` 对象。`using` 块结束时自动恢复原始 ID。不用 `using` 的话上下文不恢复，后续代码会在错误租户中执行——比如宿主查看完阳光数据后忘记恢复，后续操作可能把数据写入其他租户。

### 面试题 8（中级 / 概念题）
**题目**：多租户过滤和软删除过滤如何同时生效？

**参考答案**：两者都是 EF Core 全局查询过滤器，自动叠加。查询老人列表时实际 SQL 为 `WHERE TenantId = @xxx AND IsDeleted = 0`，同时保证属于当前养老院且未被删除。可用 `_dataFilter.Disable<ISoftDelete>()` 临时禁用软删除过滤查看已删除记录，但租户过滤通常不应禁用。

### 面试题 9（高级 / 设计题）
**题目**：大型连锁集团要求数据物理隔离，ABP 能否支持？

**参考答案**：ABP 支持独立数据库模式。在 `AbpDbConnectionOptions` 中为每个租户配置不同连接字符串，`IDbContextProvider` 根据当前租户自动选择数据库。阳光养老院数据在独立实例中，DBA 直接查询也无法看到其他养老院数据。代价是运维复杂度显著增加，每增加一个租户需要新建数据库并执行迁移。

### 面试题 10（中级 / 概念题）
**题目**：如何为不同养老院配置不同功能模块？

**参考答案**：通过 `SettingDefinitionProvider` 定义设置项（如月费、远程探视开关），管理界面或种子数据为各租户设置不同值。`ISettingProvider.GetOrNullAsync` 读取时自动返回当前租户的配置，阳光养老院月费 5000 元启用远程探视，幸福养老院月费 8000 元不启用，无需条件判断。

### 面试题 11（初级 / 概念题）
**题目**：多租户系统最大的安全风险是什么？如何防范？

**参考答案**：最大风险是数据泄漏——租户 A 访问到租户 B 的数据。例如开发者查询时忘记加 TenantId 过滤。ABP 的 `IMultiTenant` 从机制上自动添加过滤条件杜绝此风险。此外不应在 URL 中暴露租户标识，优先域名或 Header 解析。切换租户上下文时必须用 `using`，防止上下文泄漏到其他操作。

### 面试题 12（高级 / 场景题）
**题目**：描述为养老院系统添加多租户支持的完整步骤。

**参考答案**：五步：一、在 `HttpApiHostModule` 中按优先级配置域名、Header、QueryString 解析器；二、为业务实体实现 `IMultiTenant` 接口并添加 `TenantId` 属性；三、编写种子数据贡献者，用 `ICurrentTenant.Change` 切换上下文为各养老院创建初始数据；四、编写测试用例在不同租户上下文中查询验证数据隔离；五、如有个性化配置需求，通过 `SettingDefinitionProvider` 定义并为各租户设置不同值。完成后运行测试验证功能正确性。
