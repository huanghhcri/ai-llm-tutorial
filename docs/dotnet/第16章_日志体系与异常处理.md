## 为什么需要学这个？

线上出了 bug，用户说"页面报错了"，你怎么定位？如果你的系统没有日志，那就只能靠猜。更糟糕的是，很多初级开发者习惯用 `Console.WriteLine` 输出调试信息——这些信息在 IIS 部署后根本看不到，重启后更是消失得无影无踪。企业级系统需要一套**结构化、可查询、可告警**的日志体系。

---

## 1. 为什么需要结构化日志

### 1.1 养老院场景：线上 bug 怎么快速定位？

某天下午，前台护士反馈："给长者办理退住时，页面白屏了。"开发人员登录服务器，面对的问题是：这个操作涉及哪些接口？是数据库报错还是业务逻辑报错？是哪个长者？什么时间发生的？如果你只有零散的 `Console.WriteLine("出错了")`，根本无从查起。

### 1.2 传统 Console.WriteLine 的局限

| 问题 | 说明 |
|------|------|
| 无法持久化 | IIS 重启后输出消失，没有任何历史记录 |
| 无法检索 | 10 万条日志里怎么找某条特定记录？ |
| 没有级别区分 | 调试信息和严重错误看起来一模一样 |
| 没有上下文 | 不知道是哪个用户、哪个租户触发的 |

### 1.3 结构化日志的优势

结构化日志不是简单的文本，而是带字段的记录，可以被机器解析和查询：

```json
{
  "Timestamp": "2026-07-10T14:32:01.123+08:00",
  "Level": "Error",
  "Message": "退住操作失败: 张三, 原因: 存在未结算费用",
  "Properties": {
    "ElderName": "张三",
    "UserId": 42,
    "TenantId": 1
  }
}
```

这样的日志可以直接在 Kibana 里按 `Level=Error AND TenantId=1` 过滤，秒级定位问题。

## 2. Serilog 配置详解

### 2.1 安装 NuGet 包

```bash
dotnet add package Serilog.AspNetCore --version 4.1.0
dotnet add package Serilog.Sinks.File --version 5.0.0
dotnet add package Serilog.Sinks.Async --version 1.4.0
dotnet add package Serilog.Sinks.Elasticsearch --version 6.5.0
dotnet add package Serilog.Enrichers.Environment
dotnet add package Serilog.Enrichers.Thread
```

### 2.2 在 Program.cs 中配置

```csharp
public class Program
{
    public static int Main(string[] args)
    {
        Log.Logger = new LoggerConfiguration()
            .Enrich.FromLogContext()
            .Enrich.WithEnvironmentName()
            .Enrich.WithMachineName()
            .Enrich.WithThreadId()
            .WriteTo.Console()
            .WriteTo.File(
                path: "logs/nursing-home-.log",
                rollingInterval: RollingInterval.Day,
                retainedFileCountLimit: 30,
                fileSizeLimitBytes: 50 * 1024 * 1024)
            .CreateLogger();

        try
        {
            Log.Information("养老院管理系统启动中...");
            var host = CreateHostBuilder(args).Build();
            host.Run();
            return 0;
        }
        catch (Exception ex)
        {
            Log.Fatal(ex, "养老院管理系统启动失败!");
            return 1;
        }
        finally
        {
            Log.CloseAndFlush();
        }
    }

    public static IHostBuilder CreateHostBuilder(string[] args) =>
        Host.CreateDefaultBuilder(args)
            .UseSerilog()
            .ConfigureWebHostDefaults(webBuilder =>
            {
                webBuilder.UseStartup<Startup>();
            });
}
```

### 2.3 File Sink：按日期滚动

```csharp
.WriteTo.File(
    path: "logs/nursing-home-.log",
    rollingInterval: RollingInterval.Day,
    retainedFileCountLimit: 30,
    fileSizeLimitBytes: 50 * 1024 * 1024)
```

生成的文件名：`nursing-home-20260710.log`。`retainedFileCountLimit` 自动清理旧文件。

### 2.4 Console Sink

开发环境最有用，输出到控制台便于实时观察，不同级别用不同颜色高亮。

### 2.5 Elasticsearch Sink

将日志推送到 Elasticsearch，配合 Kibana 实现可视化查询：

```csharp
.WriteTo.Elasticsearch(new ElasticsearchSinkOptions(
    new Uri("http://localhost:9200"))
{
    AutoRegisterTemplate = true,
    AutoRegisterTemplateVersion = AutoRegisterTemplateVersion.ESv7,
    IndexFormat = "nursing-home-logs-{0:yyyy.MM.dd}",
    BatchPostingLimit = 50,
    Period = TimeSpan.FromSeconds(2),
    BufferBaseFilename = "logs/elasticsearch-buffer",
    CustomFormatter = new ExceptionAsObjectJsonFormatter(renderMessage: true)
})
```

`IndexFormat` 按天生成索引，方便定期清理旧数据。`BufferBaseFilename` 在 ES 不可用时缓存日志到本地文件，恢复后自动重发。

### 2.6 Async Sink：异步写入

日志写入不应阻塞业务请求，Async Sink 在内存中维护缓冲队列，后台线程批量写入：

```csharp
.WriteTo.Async(a => a.File(
    path: "logs/nursing-home-.log",
    rollingInterval: RollingInterval.Day,
    retainedFileCountLimit: 30))
```

> 注意：Async Sink 的缓冲队列默认无上限，极端情况可能占用大量内存。生产环境建议设置 `bufferSize` 参数。

## 3. 日志级别策略

### 3.1 Serilog 六个级别

| 级别 | 用途 | 示例 |
|------|------|------|
| Verbose | 最详细的追踪信息 | 循环内部变量值 |
| Debug | 调试信息，开发时有用 | 方法入参、SQL 语句 |
| Information | 业务流程关键节点 | "长者张三入住成功" |
| Warning | 不影响运行但需要关注 | "接口响应超过 3 秒" |
| Error | 操作失败但应用可继续 | "退住失败：未结算费用" |
| Fatal | 致命错误，应用无法继续 | "数据库连接池耗尽" |

### 3.2 开发 vs 生产环境配置

```json
{
  "Serilog": {
    "MinimumLevel": {
      "Default": "Information",
      "Override": {
        "Microsoft": "Warning",
        "Microsoft.Hosting.Lifetime": "Information",
        "Volo.Abp": "Information",
        "NursingHome": "Debug"
      }
    }
  }
}
```

- 开发环境：`Default` 设为 `Debug`，看到更多细节
- 生产环境：`Default` 设为 `Information`，只记录关键节点
- 框架层统一设为 `Warning`，避免框架日志淹没业务日志

生产环境将 ABP 框架日志也提升到 Warning，减少日志量和存储压力。

## 4. ABP 异常处理机制

### 4.1 三种异常类型对比

| 类型 | 显示给用户 | ABP 行为 | 适用场景 |
|------|-----------|---------|---------|
| UserFriendlyException | ✅ 直接显示消息 | 包装为 403 + 错误响应体 | 用户输入校验、业务提示（注意：不是授权异常，是业务异常） |
| BusinessException | ✅ 显示本地化消息 | 自动查找本地化文本 | 多语言业务异常 |
| 普通 Exception | ❌ 不暴露细节 | 记录日志，返回通用错误 | 系统级错误 |

### 4.2 UserFriendlyException

```csharp
public class CheckInAppService : ApplicationService
{
    public async Task CheckInAsync(CheckInDto input)
    {
        var elder = await _elderRepository.GetAsync(input.ElderId);
        if (elder.Status == ElderStatus.CheckedIn)
        {
            throw new UserFriendlyException(
                $"长者 {elder.Name} 当前已在住，不能重复入住！");
        }
        // 正常入住逻辑...
    }
}
```

前端收到响应：
```json
{
  "error": {
    "message": "长者 张三 当前已在住，不能重复入住！",
    "details": null
  }
}
```

### 4.3 BusinessException

BusinessException 适合需要国际化支持的场景：

```csharp
// 定义错误码
public static class NursingHomeErrorCodes
{
    public const string ElderAlreadyCheckedIn = "NursingHome:ElderAlreadyCheckedIn";
    public const string InsufficientBalance = "NursingHome:InsufficientBalance";
}

// 使用
throw new BusinessException("NursingHome:ElderAlreadyCheckedIn");
```

ABP 自动查找对应的本地化文本返回给前端。

### 4.4 普通 Exception

普通异常不应该暴露给前端，ABP 默认记录日志并返回通用错误消息：

```csharp
public async Task<BillDto> GenerateBillAsync(Guid elderId)
{
    try
    {
        var result = await _billingClient.CalculateAsync(elderId);
        return ObjectMapper.Map<BillResult, BillDto>(result);
    }
    catch (HttpRequestException ex)
    {
        Logger.LogError(ex, "调用计费系统失败，长者ID: {ElderId}", elderId);
        throw; // ABP 捕获并返回通用错误
    }
}
```

## 5. 全局异常过滤器

### 5.1 AbpExceptionFilter 工作原理

ABP 内置的 `AbpExceptionFilter` 核心流程：

```
请求进入 Controller → Action 执行 → 抛出异常
                                    ↓
                         AbpExceptionFilter 捕获
                                    ↓
    UserFriendlyException → 返回 403（业务异常）+ 错误消息直接显示给用户
    BusinessException → 返回 403 + 本地化消息
    EntityNotFoundException → 返回 404
    其他 Exception → 记录日志 + 返回 500 通用错误
```

### 5.2 自定义异常处理

```csharp
public class NursingHomeExceptionHandler : ExceptionHandlerBase
{
    private readonly ILogger<NursingHomeExceptionHandler> _logger;

    public NursingHomeExceptionHandler(
        ILogger<NursingHomeExceptionHandler> logger)
    {
        _logger = logger;
    }

    public override async Task HandleAsync(ExceptionHandlingContext context)
    {
        if (context.Exception is UserFriendlyException)
            return; // ABP 默认处理

        _logger.LogError(context.Exception,
            "养老院系统未处理异常: {Message}", context.Exception.Message);
    }
}
```

在模块中配置：

```csharp
Configure<AbpExceptionHandlingOptions>(options =>
{
    options.SendExceptionsDetailsToClient = false; // 生产环境不暴露详情
});
```

### 6.2 编写异常邮件中间件

```csharp
public class ExceptionEmailNotificationMiddleware : IMiddleware
{
    private readonly IEmailSender _emailSender;
    private readonly ILogger<ExceptionEmailNotificationMiddleware> _logger;

    public ExceptionEmailNotificationMiddleware(
        IEmailSender emailSender,
        ILogger<ExceptionEmailNotificationMiddleware> logger)
    {
        _emailSender = emailSender;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        try
        {
            await next(context);
        }
        catch (Exception ex) when (ShouldNotify(ex))
        {
            await SendAlertAsync(context, ex);
            throw;
        }
    }

    private bool ShouldNotify(Exception ex)
    {
        return ex is not UserFriendlyException &&
               ex is not BusinessException &&
               ex is not AbpValidationException;
    }

    private async Task SendAlertAsync(HttpContext context, Exception ex)
    {
        var body = $@"
<h2>养老院管理系统异常告警</h2>
<p><b>时间：</b>{DateTime.Now:yyyy-MM-dd HH:mm:ss}</p>
<p><b>路径：</b>{context.Request.Path}</p>
<p><b>异常：</b>{ex.GetType().Name}: {ex.Message}</p>
<pre>{ex.StackTrace}</pre>";

        try
        {
            await _emailSender.QueueAsync(
                "dev-team@nursing-home.com",
                "[养老院-告警] 系统异常通知",
                body, isBodyHtml: true);
        }
        catch (Exception mailEx)
        {
            _logger.LogError(mailEx, "发送异常告警邮件失败");
        }
    }
}
```

## 7. 审计日志详细配置与查询

### 7.1 详细配置

```csharp
Configure<AbpAuditingOptions>(options =>
{

    options.IsEnabledForAnonymousUsers = false;
    options.IsEnabledForGetRequests = false;

                type.Namespace.StartsWith("NursingHome")));

```

### 7.2 自定义审计字段

在关键业务操作中添加审计信息：

```csharp
[Audited]
public async Task CheckInAsync(CheckInDto input)
{
    // 入住逻辑...
    // ABP 自动记录操作人、时间、实体变更

```

### 7.3 查询审计日志

```csharp
public class AuditLogQueryAppService : ApplicationService
{
    private readonly IRepository<AuditLog, Guid> _auditLogRepository;

    public AuditLogQueryAppService(IRepository<AuditLog, Guid> auditLogRepository)
    {
        _auditLogRepository = auditLogRepository;
    }

    public async Task<List<AuditLogDto>> GetAuditLogsAsync(AuditLogQueryInput input)
    {
        var query = await _auditLogRepository.GetQueryableAsync();

        query = query
            .WhereIf(!string.IsNullOrEmpty(input.UserName),
                x => x.UserName == input.UserName)
            .WhereIf(input.StartTime.HasValue,
                x => x.ExecutionTime >= input.StartTime!.Value)
            .WhereIf(input.EndTime.HasValue,
                x => x.ExecutionTime <= input.EndTime!.Value)
            .OrderByDescending(x => x.ExecutionTime)
            .PageBy(input.SkipCount, input.MaxResultCount);

        var logs = await AsyncExecuter.ToListAsync(query);
        return ObjectMapper.Map<List<AuditLog>, List<AuditLogDto>>(logs);
    }
}
```

> 在 Kibana 中也可以通过 ES 索引直接查询审计日志，可视化效果更好。


### 8.4 业务代码中正确使用日志

```csharp
public class CheckInAppService : ApplicationService
{
    private readonly ILogger<CheckInAppService> _logger;
    private readonly IRepository<Elder, Guid> _elderRepository;
    private readonly IRepository<Room, Guid> _roomRepository;

    public CheckInAppService(
        ILogger<CheckInAppService> logger,
        IRepository<Elder, Guid> elderRepository,
        IRepository<Room, Guid> roomRepository)
    {
        _logger = logger;
        _elderRepository = elderRepository;
        _roomRepository = roomRepository;
    }

    public async Task CheckInAsync(CheckInDto input)
    {
        _logger.LogInformation("开始办理入住，长者ID: {ElderId}, 房间: {RoomNumber}",
            input.ElderId, input.RoomNumber);

        var elder = await _elderRepository.GetAsync(input.ElderId);
        if (elder.Status == ElderStatus.CheckedIn)
        {
            _logger.LogWarning("长者 {ElderName} 已在住，尝试重复入住", elder.Name);
            throw new UserFriendlyException($"长者 {elder.Name} 当前已在住！");
        }

        var room = await _roomRepository.GetAsync(input.RoomId);
        if (room.OccupiedBeds >= room.TotalBeds)
        {
            _logger.LogWarning("房间 {RoomNumber} 已满", room.Number);
            throw new UserFriendlyException($"房间 {room.Number} 已满！");
        }

        try
        {
            elder.Status = ElderStatus.CheckedIn;
            elder.RoomId = room.Id;
            room.OccupiedBeds++;
            await _elderRepository.UpdateAsync(elder);
            await _roomRepository.UpdateAsync(room);

            _logger.LogInformation("长者 {ElderName} 入住房间 {RoomNumber} 成功",
                elder.Name, room.Number);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "办理入住失败，长者: {ElderName}, 房间: {RoomNumber}",
                elder.Name, room.Number);
            throw;
        }
    }
}
```

### 8.5 关键要点总结

| 要点 | 说明 |
|------|------|
| 日志结构化 | 用 `{Placeholder}` 不用字符串拼接，Serilog 自动提取为结构化字段 |
| 级别分明 | 业务成功 Information，异常 Error，调试 Debug |
| 异常分类 | 用户可理解的用 UserFriendlyException，系统错误走普通 Exception |
| 多 Sink 分发 | 本地文件做备份，ES 做集中查询，Console 做容器日志 |
| 异步写入 | Async Sink 避免日志 IO 阻塞业务请求 |
| 异常告警 | Error 级别异常发邮件，第一时间发现线上问题 |
| 审计日志 | 关键操作自动记录，可追溯、可审计 |

这套方案覆盖了从开发调试到生产监控的完整链路，是企业级 .NET 项目的标准日志实践。

## 常见错误与最佳实践

| # | 错误写法 | 正确写法 | 为什么错 |
|---|---------|---------|---------|
| 1 | 生产环境用 Debug 级别日志 | 生产用 Information/Warning，开发用 Debug | Debug 日志量巨大，磁盘很快写满，且暴露内部逻辑 |
| 2 | `catch (Exception ex) { }` 空 catch | 至少记录日志 `Log.Error(ex, "消息")` | 异常被吞掉，出了 bug 无任何线索排查 |
| 3 | 抛出 `new Exception("业务错误")` | 抛出 `UserFriendlyException("业务错误")` | 普通 Exception 被全局过滤器捕获返回 500，用户看到"系统错误" |
| 4 | 日志消息用字符串拼接 `$"用户{id}"` | 用结构化日志 `Log.Information("用户{UserId}", id)` | 拼接无法被 ES 索引，结构化日志可按字段搜索 |
| 5 | 所有异常都发邮件通知 | 只有 Error/Fatal 级别才发邮件 | Warning 级别（如配置缺失但有默认值）不应打扰开发团队 |
| 6 | 日志文件不限大小和数量 | 配置 `rollingInterval` + `retainedFileCountLimit` | 日志文件无限增长最终磁盘满，服务器宕机 |
| 7 | 敏感信息写入日志（密码、Token） | 日志中脱敏处理，不记录敏感字段 | 日志可能被未授权人员查看，造成数据泄露 |
| 8 | 不配置审计日志 | 生产环境启用 ABP 审计日志 | 出了安全事件无法追溯"谁在什么时间做了什么" |

---

## 本章小结

本章学习了企业级日志体系与异常处理：**Serilog** 是结构化日志引擎，通过 File/Console/ES 三种 Sink 覆盖本地存储、实时搜索、可视化分析需求；**日志级别**按环境区分（开发 Debug、生产 Information）；**ABP 异常分类**——`UserFriendlyException` 直接显示给用户（业务提示）、`BusinessException` 支持多语言（国际化场景）、普通 `Exception` 记录日志不暴露细节（系统异常）；**全局异常过滤器**统一处理所有未捕获异常；**审计日志**自动记录操作人/时间/参数/返回值，实体变更追踪字段级修改历史。核心原则：**结构化日志便于搜索，异常分类便于处理，审计日志便于追溯。**

## 面试题

### 面试题 1（初级 / 概念题）
**题目**：Serilog 的六个日志级别分别是什么？各用于什么场景？

**参考答案**：**Verbose**（最详细，开发调试用）、**Debug**（调试信息，开发环境用）、**Information**（常规业务流程记录，生产环境默认级别）、**Warning**（异常但不影响业务，如配置缺失但有默认值）、**Error**（异常导致操作失败，需要关注）、**Fatal**（致命错误，系统可能无法继续运行）。养老院场景：长者入住成功用 Information，入住人数超限用 Warning，账单生成失败用 Error。

### 面试题 2（中级 / 概念题）
**题目**：UserFriendlyException 和 BusinessException 有什么区别？

**参考答案**：`UserFriendlyException` 的消息会直接显示给前端用户（如"该长者已入住，不能重复入住"），通常不写入 Error 级别系统日志；但操作仍可能出现在审计日志（AuditLog）中。`BusinessException` 是业务异常，支持多语言（通过错误码查找本地化消息），ABP 会记录审计日志。普通 `Exception` 是系统异常，ABP 全局过滤器捕获后返回 500，不暴露内部细节。养老院场景：业务校验用 `UserFriendlyException`，需要多语言的用 `BusinessException`。

### 面试题 3（初级 / 概念题）
**题目**：什么是结构化日志？和传统字符串日志有什么区别？

**参考答案**：传统日志 `Log($"用户{id}在{time}入住")` 输出纯字符串，无法按字段搜索。结构化日志 `Log.Information("用户{UserId}在{Time}入住", id, time)` 输出结构化数据，ES 可以按 `UserId`、`Time` 字段索引和搜索。养老院场景：需要查"某个护工今天操作了哪些长者"，结构化日志可以按 `UserId` 字段精确搜索，字符串日志只能全文模糊匹配。

### 面试题 4（中级 / 概念题）
**题目**：Serilog 的 File Sink 如何配置按日期滚动？

**参考答案**：配置 `rollingInterval: RollingInterval.Day` 按天滚动，`retainedFileCountLimit: 30` 保留最近 30 天日志。文件名自动附加日期（如 `log-20260710.txt`）。还可以配置 `fileSizeLimitBytes` 限制单文件大小。养老院场景：生产环境每天生成一个日志文件，保留 30 天，超大日志自动分割。

### 面试题 5（高级 / 场景题）
**题目**：如何配置生产环境的异常邮件通知？

**参考答案**：在 Serilog 的 `LoggerFilter` 或自定义中间件中，当日志级别 ≥ Error 时触发邮件发送。注入 `IEmailSender`（ABP 内置），将异常信息（时间、请求路径、用户、堆栈）格式化为邮件正文，发送给开发团队。注意：① 只发 Error/Fatal，不发 Warning；② 加频率限制（如同一异常 5 分钟内只发一次），避免邮件轰炸。养老院场景：账单生成接口报错，5 分钟内收到一封邮件。

### 面试题 6（中级 / 概念题）
**题目**：ABP 的 AbpExceptionFilter 工作流程是什么？

**参考答案**：当 Controller 方法抛出未处理异常时：① 判断异常类型——`UserFriendlyException` 返回 403 + 消息体；`EntityNotFoundException` 返回 404；`AbpAuthorizationException` 返回 401/403；其他 `Exception` 返回 500 + 通用错误消息（不暴露细节）。② 所有异常都记录审计日志。③ 开发环境可配置显示详细异常信息。养老院场景：护工调用未授权接口返回 403，系统 bug 返回 500。

### 面试题 7（初级 / 概念题）
**题目**：为什么生产环境不应该用 Debug 级别日志？

**参考答案**：Debug 日志包含大量内部调试信息（SQL 语句、变量值、方法调用链），日志量是 Information 的 10-100 倍。会导致：① 磁盘快速写满；② ES 索引压力大；③ 真正的错误信息被淹没在海量调试日志中。养老院场景：生产环境用 Information 级别，只记录业务流程（入住、缴费、护理）；开发环境用 Debug 级别，方便调试。

### 面试题 8（高级 / 场景题）
**题目**：如何在日志中追踪一个请求的完整链路？

**参考答案**：使用 `CorrelationId`（关联 ID）。每个请求生成唯一 ID（如 GUID），通过中间件注入到 `HttpContext.Items`，日志中所有输出都包含这个 ID。跨服务调用时通过 Header 传递。养老院场景：长者入住请求涉及多个服务（长者服务→财务服务→通知服务），通过 CorrelationId 可以串联所有日志，快速定位是哪个环节出错。

### 面试题 9（中级 / 概念题）
**题目**：ABP 审计日志默认记录哪些信息？

**参考答案**：操作人（UserId/UserName/TenantId）、操作时间、执行耗时、服务名、方法名、HTTP 方法和路径、请求参数、返回值、客户端 IP、浏览器信息、异常信息。启用实体变更追踪后还记录字段级变更（属性名、旧值、新值）。养老院场景：修改长者护理等级时，审计日志记录"张三于 2026-07-10 将护理等级从自理改为半护理"。

### 面试题 10（高级 / 设计题）
**题目**：如何设计一个生产环境的完整日志方案？

**参考答案**：三层架构：① **Serilog File Sink**——本地日志文件，按天滚动，保留 30 天，作为最后保底；② **Serilog Elasticsearch Sink**——实时写入 ES，配合 Kibana 做可视化搜索和告警；③ **异常邮件通知**——Error 级别触发邮件。配置 `Serilog.Sinks.Async` 异步写入，避免日志 IO 阻塞业务线程。养老院系统：日常排查用 Kibana 搜索，紧急问题靠邮件通知，磁盘日志作为兜底。
