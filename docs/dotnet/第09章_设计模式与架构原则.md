## 为什么需要学这个？

设计模式和架构原则是 .NET 面试中出现频率最高的话题之一。面试官问"说说 SOLID"，不是要你背定义，而是要你能结合实际项目讲清楚每个原则如何落地。ABP 框架本身就是设计模式的集大成者——仓储模式、策略模式、观察者模式在 ABP 中无处不在。理解这些模式，不仅帮你读懂 ABP 源码，更能写出可维护的企业级代码。

## 1. SOLID 五大原则

### 1.1 生活类比：养老院管理的五大原则

| 原则 | 养老院类比 |
|------|-----------|
| **S 单一职责** | 护士只管护理，厨师只管做饭，不能让护士既打针又炒菜 |
| **O 开闭原则** | 新增特护病房，不需要拆掉现有普通病房 |
| **L 里氏替换** | 任何正式护工都能替代值班护工，老人不受影响 |
| **I 接口隔离** | 老人只需"护理服务"，不依赖"厨房管理系统" |
| **D 依赖倒置** | 院长对接"护理部接口"，不直接指挥每个护工 |

### 1.2 S — 单一职责原则（SRP）

一个类只有一个引起它变化的原因。

**反面教材**：`ElderAppService` 既管理 CRUD 又发通知邮件：

```csharp
// ❌ 违反 SRP
public class ElderAppService : ApplicationService
{
    private readonly IRepository<Elder, Guid> _repository;
    private readonly IEmailSender _emailSender;

    public async Task<ElderDto> CreateAsync(CreateElderDto input)
    {
        var elder = ObjectMapper.Map<CreateElderDto, Elder>(input);
        await _repository.InsertAsync(elder);
        await _emailSender.SendAsync("admin@home.com", "入住通知", elder.Name);
        return ObjectMapper.Map<Elder, ElderDto>(elder);
    }
}
```

**正确做法**——拆分为两个类：

```csharp
// ✅ ElderAppService 只负责 CRUD
public class ElderAppService :
    CrudAppService<Elder, ElderDto, Guid, PagedAndSortedResultRequestDto>,
    IElderAppService
{
    public ElderAppService(IRepository<Elder, Guid> repo) : base(repo) { }
}

// ✅ 通知服务只负责发消息
public class ElderNotificationService : ITransientDependency
{
    private readonly IEmailSender _emailSender;
    public ElderNotificationService(IEmailSender emailSender)
        => _emailSender = emailSender;

    public async Task NotifyNewElderAsync(string name)
        => await _emailSender.SendAsync("admin@home.com", "入住通知", name);
}
```

### 1.3 O — 开闭原则（OCP）

对扩展开放，对修改关闭。

**反面教材**：新增护理等级就要改 `CalculateFee`：

```csharp
// ❌ 每加一种等级就要改这个方法
public decimal CalculateFee(string level, int days) => level switch
{
    "自理" => days * 100m,
    "半护理" => days * 200m,
    "全护理" => days * 350m,
    _ => throw new ArgumentException("未知等级")
};
```

**正确做法**——定义接口，新等级只需新增类：

```csharp
// ✅ 策略接口
public interface INursingFeeStrategy
{
    string Level { get; }
    decimal Calculate(int days);
}

public class SelfCareFeeStrategy : INursingFeeStrategy
{
    public string Level => "自理";
    public decimal Calculate(int days) => days * 100m;
}

// 新增特护？只需添加新类，不修改任何已有代码
public class SpecialCareFeeStrategy : INursingFeeStrategy
{
    public string Level => "特护";
    public decimal Calculate(int days) => days * 500m;
}
```

### 1.4 L — 里氏替换原则（LSP）

子类对象可以替换父类对象，程序行为不变。

```csharp
// ✅ 所有 AppService 都能当 IApplicationService 用
public class ElderAppService : CrudAppService<Elder, ElderDto, Guid, ...> { }
public class BedAppService : CrudAppService<Bed, BedDto, Guid, ...> { }

// ABP 框架不需要知道具体是哪个 AppService
// 任何 IApplicationService 都能被其子类替换
```

### 1.5 I — 接口隔离原则（ISP）

不应强迫客户端依赖它不使用的接口。

```csharp
// ✅ 只需查询——注入只读仓储
public class ElderQueryService : ApplicationService
{
    private readonly IReadOnlyRepository<Elder, Guid> _repo;
    public ElderQueryService(IReadOnlyRepository<Elder, Guid> repo)
        => _repo = repo;
}

// ✅ 需要增删改——注入完整仓储
public class ElderManageService : ApplicationService
{
    private readonly IRepository<Elder, Guid> _repo;
    public ElderManageService(IRepository<Elder, Guid> repo)
        => _repo = repo;
}
```

### 1.6 D — 依赖倒置原则（DIP）

高层模块不应依赖低层模块，两者都应依赖抽象。

```csharp
// ❌ 依赖具体实现
public ElderAppService(EfCoreElderRepository repo) { }

// ✅ 依赖抽象接口——换 ORM 时应用层零改动
public ElderAppService(IRepository<Elder, Guid> repo) { }
```

## 2. 常见设计模式在 ABP 项目中的体现

### 2.1 仓储模式（Repository）

```csharp
public class ElderAppService : ApplicationService
{
    private readonly IRepository<Elder, Guid> _repo;
    public ElderAppService(IRepository<Elder, Guid> repo) => _repo = repo;

    public async Task<List<ElderDto>> GetByFloorAsync(int floor)
    {
        var list = await _repo.GetListAsync(e => e.Floor == floor);
        return ObjectMapper.Map<List<Elder>, List<ElderDto>>(list);
    }
}
```

### 2.2 工厂模式（Factory）

```csharp
// ABP 内置工厂方法
var bed = new Bed(GuidGenerator.Create(), "A101", 1);

// 自定义工厂：根据配置创建短信客户端
public class SmsClientFactory : ISmsClientFactory, ITransientDependency
{
    private readonly IConfiguration _config;
    public SmsClientFactory(IConfiguration config) => _config = config;

    public ISmsClient Create() => _config["Sms:Provider"] switch
    {
        "Aliyun" => new AliyunSmsClient(_config),
        "Tencent" => new TencentSmsClient(_config),
        _ => throw new ArgumentException("不支持的供应商")
    };
}
```

### 2.3 策略模式（Strategy）

```csharp
public class FeeCalculator : ITransientDependency
{
    private readonly IEnumerable<INursingFeeStrategy> _strategies;
    public FeeCalculator(IEnumerable<INursingFeeStrategy> strategies)
        => _strategies = strategies;

    public decimal Calculate(string level, int days)
    {
        var s = _strategies.FirstOrDefault(x => x.Level == level)
            ?? throw new BusinessException("UnknownLevel");
        return s.Calculate(days);
    }
}
```

### 2.4 观察者模式（Observer）

```csharp
// 事件载体
public class ElderCheckedInEto
{
    public Guid ElderId { get; set; }
    public string ElderName { get; set; }
}

// 处理器1：生成账单
public class BillingHandler : IDistributedEventHandler<ElderCheckedInEto>
{
    public async Task HandleEventAsync(ElderCheckedInEto e)
        => Console.WriteLine($"为 {e.ElderName} 生成账单");
}

// 处理器2：分配床位
public class BedHandler : ILocalEventHandler<ElderCheckedInEto>
{
    public async Task HandleEventAsync(ElderCheckedInEto e)
        => Console.WriteLine($"为 {e.ElderName} 分配床位");
}

// 发布者不知道谁在监听
await _eventBus.PublishAsync(new ElderCheckedInEto { ElderId = id, ElderName = "张大爷" });
```

### 2.5 模板方法模式（Template Method）

```csharp
// CrudAppService 是模板方法——重写需要定制的步骤
public class ElderAppService :
    CrudAppService<Elder, ElderDto, Guid, PagedAndSortedResultRequestDto>
{
    public ElderAppService(IRepository<Elder, Guid> repo) : base(repo) { }

    protected override async Task<Elder> MapToEntityAsync(CreateElderDto input)
    {
        var entity = await base.MapToEntityAsync(input);
        if (entity.Age < 60) throw new BusinessException("Elder:AgeRequirement");
        return entity;
    }

    protected override IQueryable<Elder> CreateFilteredQuery(
        PagedAndSortedResultRequestDto input)
        => base.CreateFilteredQuery(input).Where(e => e.Status == ElderStatus.Active);
}

// BackgroundWorker 也是模板方法
public class DailyFeeWorker : AsyncPeriodicBackgroundWorkerBase
{
    public DailyFeeWorker(AbpAsyncTimer timer, IServiceScopeFactory sf)
        : base(timer, sf) { Timer.Period = 86400000; }

    protected override async Task DoWorkAsync(PeriodicBackgroundWorkerContext ctx)
    {
        var svc = ctx.ServiceProvider.GetRequiredService<IFeeAppService>();
        await svc.CalculateDailyFeesAsync();
    }
}
```

### 2.6 装饰器模式（Decorator）

ASP.NET Core 中间件管道就是装饰器——每个中间件装饰下一个：

```csharp
public class VisitorLoggingMiddleware : IMiddleware
{
    private readonly ILogger<VisitorLoggingMiddleware> _logger;
    public VisitorLoggingMiddleware(ILogger<VisitorLoggingMiddleware> logger)
        => _logger = logger;

    public async Task InvokeAsync(HttpContext ctx, RequestDelegate next)
    {
        _logger.LogInformation("访客进入: {Path}", ctx.Request.Path);
        await next(ctx);  // 调用下一个"装饰器"
        _logger.LogInformation("访客离开: {Code}", ctx.Response.StatusCode);
    }
}
```

## 3. DDD 核心概念深化

### 3.1 聚合根（AggregateRoot）

```csharp
public class Elder : AggregateRoot<Guid>
{
    public string Name { get; private set; }
    public string NursingLevel { get; private set; }

    private readonly List<NursingRecord> _records;
    public IReadOnlyCollection<NursingRecord> Records => _records.AsReadOnly();

    protected Elder() { }
    public Elder(Guid id, string name) : base(id)
    {
        Name = name;
        _records = new List<NursingRecord>();
    }

    // 只有通过聚合根才能添加记录——保证业务规则
    public NursingRecord AddRecord(string content, string nurse)
    {
        if (NursingLevel == null)
            throw new BusinessException("Elder:NursingLevelRequired");
        var r = new NursingRecord(GuidGenerator.Create(), Id, content, nurse);
        _records.Add(r);
        return r;
    }
}
```

聚合边界：内部强一致性（事务），聚合之间通过 ID 引用 + 领域事件最终一致。

### 3.2 限界上下文（BoundedContext）

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ 长者管理上下文 │  │ 财务管理上下文 │  │ 健康管理上下文 │
│ Elder        │  │ Bill         │  │ HealthRecord │
│ Bed          │  │ Payment      │  │ Medication   │
│ NursingRecord│  │ FeeStandard  │  │ Checkup      │
└──────────────┘  └──────────────┘  └──────────────┘
```

每个上下文是一个独立 ABP Module，通过领域事件跨上下文通信。

### 3.3 领域事件驱动设计


## 4. 实战：用策略模式重构护理等级评估

### 4.1 问题：if-else 嵌套太深

```csharp
// ❌ 重构前
public string Evaluate(Elder e)
{
    if (e.CanWalk)
    {
        if (e.CanSelfCare)
        {
            if (e.HasChronicDisease)
            {
                if (e.NeedsMedication) return "半护理-慢性病";
                return "自理-观察";
            }
            return "自理";
        }
        else { return e.NeedsMedication ? "全护理" : "半护理"; }
    }
    else
    {
        if (e.CanSelfCare) return "半护理";
        return e.Age >= 80 ? "特护" : "全护理";
    }
}
```

### 4.2 方案：策略模式重构

**第一步：定义接口**

```csharp
public interface INursingLevelStrategy
{
    bool CanHandle(Elder elder);
    NursingEvaluationResult Evaluate(Elder elder);
}

public class NursingEvaluationResult
{
    public string Level { get; set; }
    public string Remark { get; set; }
    public decimal MonthlyFee { get; set; }
}
```

**第二步：实现策略**

```csharp
public class SelfCareStrategy : INursingLevelStrategy
{
    public bool CanHandle(Elder e) => e.CanWalk && e.CanSelfCare && !e.HasChronicDisease;
    public NursingEvaluationResult Evaluate(Elder e) => new()
    {
        Level = "自理", Remark = "身体良好", MonthlyFee = 3000m
    };
}

public class SemiCareStrategy : INursingLevelStrategy
{
    public bool CanHandle(Elder e) => e.CanWalk && e.HasChronicDisease;
    public NursingEvaluationResult Evaluate(Elder e) => new()
    {
        Level = "半护理", Remark = "有慢性病需用药指导", MonthlyFee = 5000m
    };
}

public class FullCareStrategy : INursingLevelStrategy
{
    public bool CanHandle(Elder e) => !e.CanWalk && e.CanSelfCare;
    public NursingEvaluationResult Evaluate(Elder e) => new()
    {
        Level = "全护理", Remark = "行动不便需全面护理", MonthlyFee = 8000m
    };
}

public class SpecialCareStrategy : INursingLevelStrategy
{
    public bool CanHandle(Elder e) => !e.CanWalk && !e.CanSelfCare && e.Age >= 80;
    public NursingEvaluationResult Evaluate(Elder e) => new()
    {
        Level = "特护", Remark = "高龄无法自理需24小时护理", MonthlyFee = 12000m
    };
}
```

**第三步：策略管理器**

```csharp
public class NursingLevelEvaluator : ITransientDependency
{
    private readonly IEnumerable<INursingLevelStrategy> _strategies;

    public NursingLevelEvaluator(IEnumerable<INursingLevelStrategy> strategies)
        => _strategies = strategies;

    public NursingEvaluationResult Evaluate(Elder elder)
    {
        var strategy = _strategies.FirstOrDefault(s => s.CanHandle(elder))
            ?? throw new BusinessException("Elder:NoMatchingStrategy");
        return strategy.Evaluate(elder);
    }
}
```

**第四步：在应用服务中使用**

```csharp
public class ElderEvaluationAppService : ApplicationService
{
    private readonly NursingLevelEvaluator _evaluator;
    private readonly IRepository<Elder, Guid> _repo;

    public ElderEvaluationAppService(
        NursingLevelEvaluator evaluator, IRepository<Elder, Guid> repo)
    {
        _evaluator = evaluator;
        _repo = repo;
    }

    public async Task<NursingEvaluationResult> EvaluateAsync(Guid elderId)
    {
        var elder = await _repo.GetAsync(elderId);
        return _evaluator.Evaluate(elder);
    }
}
```

重构效果：每种等级独立类，新增等级不改已有代码（OCP），所有策略可互换（LSP），每个策略可单独测试（SRP）。

## 常见错误与最佳实践

| # | 错误写法 | 正确写法 | 为什么错 |
|---|---------|---------|---------|
| 1 | 一个 AppService 上千行，什么都往里塞 | 按职责拆分为独立 Service（ElderService + NotificationService） | 违反单一职责，改一个功能要读懂整个文件 |
| 2 | 新增护理等级必须改已有 if-else | 用策略模式 + DI，新增实现类即可 | 违反开闭原则，每次新增功能都要改已有代码 |
| 3 | 查询服务注入完整 IRepository | 注入 `IReadOnlyRepository`，只暴露读操作 | 违反接口隔离，查询服务不应该有 Delete 方法 |
| 4 | 直接 `new EfCoreElderRepository()` | 通过构造函数注入 `IRepository<Elder>` | 违反依赖倒置，高层模块依赖了具体实现 |
| 5 | 外部直接操作聚合内实体的仓储 | 通过聚合根方法操作内部实体 | 破坏聚合一致性边界，业务规则可能被绕过 |
| 6 | 事件处理器之间有顺序依赖 | 用 Saga/工作流模式编排执行顺序 | 观察者模式不保证执行顺序，依赖顺序会出 bug |
| 7 | 策略 CanHandle 有重叠 | 加优先级属性或确保互斥匹配 | 多个策略同时匹配导致执行结果不确定 |
| 8 | CrudAppService 不重写 MapToEntityAsync | 重写 MapToEntityAsync 过滤敏感字段 | 默认映射可能暴露内部字段（如密码哈希、内部标记） |

## 本章小结

1. **SOLID 不是口号**，ABP 框架本身就是 SOLID 的教科书级实现
2. **设计模式要活学活用**，ABP 的 Repository、EventBus、CrudAppService 已帮你实现大部分模式
3. **DDD 的聚合根和限界上下文**帮助划分业务边界，领域事件实现松耦合
4. **策略模式是消除 if-else 的利器**，配合 DI 容器优雅实现开闭原则

## 面试题

### 面试题 1（高级 / 设计题）
**题目**：请解释 SOLID 五大原则，每个举一个养老院系统的例子。

**参考答案**：**S（单一职责）**：`ElderAppService` 只负责长者 CRUD，通知家属由 `NotificationService` 负责，两个类各管一件事。**O（开闭原则）**：新增护理等级只需添加 `INursingFeeStrategy` 实现类，不修改已有评估代码。**L（里氏替换）**：`ElderAppService` 和 `BedAppService` 都继承 `ApplicationService`，任何需要 `IApplicationService` 的地方都能替换使用。**I（接口隔离）**：只读查询注入 `IReadOnlyRepository<Elder>`，管理功能注入完整 `IRepository<Elder>`，不让查询服务依赖写方法。**D（依赖倒置）**：AppService 注入 `IRepository<Elder, Guid>` 接口而非 `EfCoreElderRepository` 具体类。

### 面试题 2（中级 / 概念题）
**题目**：ABP 中的仓储模式解决了什么问题？

**参考答案**：仓储模式封装数据访问细节，使应用服务不依赖具体 ORM。AppService 只调用 `IRepository<T>` 的 GetAsync/InsertAsync/UpdateAsync 方法，不关心底层是 EF Core 还是 Dapper。换 ORM 时只需替换 Infrastructure 层的仓储实现，业务层代码零修改。养老院场景：长者管理模块的 `ElderAppService` 注入 `IRepository<Elder>`，如果未来从 EF Core 切换到 Dapper，只需新建 `DapperElderRepository` 实现同一接口。

### 面试题 3（中级 / 概念题）
**题目**：ILocalEventHandler 和 IDistributedEventHandler 的区别？

**参考答案**：`ILocalEventHandler` 同进程内执行，事件通过内存传递，适合同一服务内的解耦（如长者入住后分配床位）；`IDistributedEventHandler` 跨服务通过消息队列（RabbitMQ/CAP）传递，适合跨上下文的异步处理（如长者入住后生成财务账单）。本地事件性能高但不能跨服务，分布式事件可靠但有延迟。养老院系统中，入住事件同时触发两种处理器：本地分配床位、分布式生成账单。

### 面试题 4（高级 / 设计题）
**题目**：如何用策略模式消除 if-else？

**参考答案**：定义策略接口 `INursingFeeStrategy`，每个护理等级实现一个策略类（`SelfCareStrategy`、`SemiCareStrategy` 等），每个类包含 `CanHandle` 判断条件和 `Calculate` 计算逻辑。通过 DI 注入 `IEnumerable<INursingFeeStrategy>`，运行时遍历找到匹配的策略执行。新增护理等级只需添加新实现类，不修改调用端代码，完美符合开闭原则。ABP 的 Autofac 支持 `IEnumerable<T>` 自动注入。

### 面试题 5（中级 / 概念题）
**题目**：CrudAppService 体现了哪种设计模式？

**参考答案**：模板方法模式。CrudAppService 定义了 CRUD 的标准流程模板（Get → 查询实体 → 映射 DTO → 返回），子类通过重写 `MapToEntityAsync`（自定义映射逻辑）、`CreateFilteredQueryAsync`（添加过滤条件）、`GetPolicyName`（权限策略）等钩子方法定制行为，而不改变整体流程。养老院场景：`ElderAppService` 继承 CrudAppService，重写 `CreateFilteredQueryAsync` 添加按护理等级过滤的条件。

### 面试题 6（初级 / 概念题）
**题目**：什么是聚合根？为什么需要它？

**参考答案**：聚合根是一致性边界的核心实体，外部只能通过聚合根访问其内部实体。例如 `Elder` 是聚合根，`NursingRecord`（护理记录）是聚合内的实体，外部不能直接操作 `NursingRecord` 的仓储，必须通过 `elder.AddRecord()` 方法添加。这样保证了业务规则（如"每次护理记录必须关联长者"）的一致性。没有聚合根，内部实体可能被任意修改，导致数据不一致。

### 面试题 7（初级 / 概念题）
**题目**：中间件管道如何体现装饰器模式？

**参考答案**：每个中间件包装下一个中间件，添加前后置行为。异常处理中间件在 `next()` 前 try-catch、在 `next()` 后检查异常；认证中间件在 `next()` 前验证 Token；日志中间件在 `next()` 前记录开始时间、在 `next()` 后记录耗时。这些中间件层层叠加，每个都为管道增加一层职责，这就是装饰器模式的典型应用。养老院系统的请求管道：CORS → 认证 → 授权 → 异常处理 → 操作日志 → Action。

### 面试题 8（初级 / 概念题）
**题目**：ABP 的 GuidGenerator 是什么设计模式？

**参考答案**：工厂模式。`GuidGenerator.Create()` 封装了 GUID 的生成逻辑，支持生成顺序 GUID（Sequential GUID），避免随机 GUID 导致的数据库 B+Tree 索引碎片化问题。InnoDB 的聚簇索引按主键排序插入，随机 GUID 会导致频繁的页分裂，而顺序 GUID 保证单调递增，插入性能更好。养老院系统中所有实体的 ID 都应使用 `GuidGenerator.Create()` 生成。

### 面试题 9（中级 / 场景题）
**题目**：如何判断一个类是否违反单一职责？

**参考答案**：问"这个类有几种原因会导致它被修改？"如果 `ElderAppService` 因长者字段变化、邮件模板变化、报表格式变化都要修改，就说明它承担了太多职责，应拆分。养老院场景：`ElderAppService` 只负责长者 CRUD，发邮件由 `NotificationService` 负责，生成报表由 `ReportService` 负责。每个类只因一种业务变化而修改，就是单一职责。

### 面试题 10（中级 / 概念题）
**题目**：限界上下文和 ABP Module 的关系？

**参考答案**：每个限界上下文通常对应一个 ABP Module。养老院系统的三个限界上下文：`ElderManagementModule`（长者管理，含 Elder/Room/Bed 实体）、`FinanceModule`（财务管理，含 Bill/Payment 实体）、`HealthModule`（健康管理，含 HealthAssessment/NursingRecord 实体）。每个 Module 有独立的实体、服务和仓储，通过领域事件（如 `ElderCheckedInEto`）跨上下文通信。Module 之间不直接引用对方的实体。

### 面试题 11（初级 / 代码题）
**题目**：在 ABP 中如何注册多个策略实现？

**参考答案**：ABP 的 Autofac 支持 `IEnumerable<T>` 自动注入。步骤：①定义策略接口 `INursingFeeStrategy`；②每个策略类实现接口并标记 `ITransientDependency`（如 `SelfCareStrategy : INursingFeeStrategy, ITransientDependency`）；③在消费端构造函数注入 `IEnumerable<INursingFeeStrategy>`；④运行时遍历找到匹配策略执行。新增策略只需添加新类，无需修改任何已有代码。

### 面试题 12（中级 / 概念题）
**题目**：领域事件和集成事件的区别？

**参考答案**：领域事件（`ILocalEventHandler`）同进程内执行，通过内存事件总线传递，性能高但不能跨服务，适合同一上下文内的解耦（如长者入住后分配床位）。集成事件（`IDistributedEventHandler`）跨进程通过消息队列（RabbitMQ + CAP）传递，可靠但有延迟，适合不同上下文间的通信（如长者入住后生成财务账单）。养老院系统中，入住事件同时触发本地事件（分配床位）和集成事件（生成账单）。

### 面试题 13（初级 / 概念题）
**题目**：如何在单元测试中利用依赖倒置？

**参考答案**：应用服务依赖 `IRepository<Elder, Guid>` 接口而非具体仓储实现，测试时用 NSubstitute/Moq 模拟接口返回值，无需启动真实数据库。例如测试入住逻辑：`_elderRepo.GetAsync(id).Returns(elder)` 模拟仓储返回，`_billRepo.When(x => x.InsertAsync(Arg.Any<Bill>())).Do(x => capturedBill = x.Arg<Bill>())` 捕获生成的账单。这样测试速度快、不依赖外部环境、专注于验证业务逻辑。

### 面试题 14（高级 / 设计题）
**题目**：装饰器模式和代理模式的区别？

**参考答案**：装饰器侧重动态添加职责（日志、缓存、重试），可叠加多层装饰器形成处理链；代理侧重控制访问（权限检查、延迟加载、远程调用封装），通常一对一。ABP 中间件管道是装饰器模式（每个中间件装饰下一个），ABP 的懒加载代理（如 `IRepository` 的动态代理）是代理模式。养老院场景：HTTP 调用封装用代理模式（控制超时和重试），日志记录用装饰器模式（在调用前后添加日志）。
